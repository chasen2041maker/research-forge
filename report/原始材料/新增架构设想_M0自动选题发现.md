# 新增架构设想：M0 自动选题发现模块

## 背景

当前项目的主流程是从用户输入的研究方向开始：

```text
用户原始想法
→ M1 问题精炼
→ M2 多源文献检索
→ M3 知识图谱构建
→ M4 多 Reviewer 批判
→ M5 实验方案设计
→ M6 代码生成与验证
→ M7 论文草稿生成
```

也就是说，现有系统更像是一个“科研方向增强器 / 课题打磨器”。它能围绕用户给定方向检索文献、整理知识、批判风险并生成实验方案，但还不是完全自动的“开放域科研选题发现系统”。

为了让系统具备更强的主动性，可以在 M1 之前新增一个 **M0 自动选题发现模块**。

---

## 核心想法

新增 M0 模块，让系统在没有明确课题输入时，能够自动扫描近期文献、识别热点趋势、发现潜在研究空白，并提出多个候选科研方向。

当前阶段不追求覆盖所有学科，建议先限定为三个目标领域：

```text
1. 计算机科学 / AI
2. 医疗 AI / 医学信息学
3. 生物信息学 / 计算生物学
```

这样可以避免开放域选题过于发散，也方便为不同领域设计专门的 M4 Reviewer、M5 实验模板和 M6 验证产物。

同时，模型调用层建议从当前 DeepSeek / Claude 直连方式，升级为全部通过中转站统一接入 GPT 与 Claude：

```text
常规生成、抽取、重排、写作：GPT 系列模型 via 中转站
复杂推理、M4 普通 Reviewer、M5 实验设计：GPT 高推理模型 via 中转站
最终裁决、关键审稿、Meta Reviewer：Claude Opus via 中转站
```

这样可以在不大改业务模块的前提下，保留 `chat / reasoner / critical` 语义角色，同时把 provider 鉴权、限流、模型替换、降级和成本统计统一收敛到中转站与 LLM factory 层；后续架构不再涉及 DeepSeek 作为默认路径。

新的整体流程变成：

```text
项目启动
→ 询问用户是否已有研究方向
→ 如果已有方向：跳过 M0，直接进入 M1 问题精炼
→ 如果没有方向：进入 M0 自动选题发现，让用户选择候选课题
→ M1 问题精炼
→ M2 多源文献检索
→ M3 知识图谱构建
→ M4 多 Reviewer 批判
→ M5 实验方案设计
→ M6 代码生成与验证
→ M7 论文草稿生成
```

其中：

```text
M0 = 找方向
M1 = 定义问题
M2 = 找证据
M3 = 整理知识
M4 = 批判风险
M5 = 落实验方案
```

## 启动时的交互分流

为了兼容两类使用场景，项目启动时可以先询问用户是否已经有明确研究方向。

```text
你是否已有明确研究方向？

1. 有，我已经知道大方向
2. 没有具体方向，但我有大致领域
3. 完全没有方向，希望系统先帮我选择领域和候选课题
```

### 情况 A：用户已有方向

如果用户已经有方向，例如：

```text
我想做医学 RAG 幻觉抑制
```

则系统跳过 M0，直接进入现有主流程：

```text
用户方向
→ M1 问题精炼
→ M2 文献检索
→ M3 知识图谱
→ M4 批判
→ M5 实验方案
```

这种模式适合：

- 用户已经有毕业设计/论文方向
- 用户已经指定领域，例如医学 NLP、RAG、Agent、知识图谱
- 用户只是希望系统帮忙打磨课题和实验方案

### 情况 B：用户没有方向

如果用户没有明确方向，但有大致领域，则先让用户选择一级领域和可选二级子领域，再进入 M0。

不建议让 M0 从完全开放域开始搜索，因为范围过大时会导致：

- 检索成本过高
- 候选课题过于分散
- 趋势判断不稳定
- 用户选择困难
- 生成的课题可能不符合用户背景

因此，M0 最好至少需要一个一级领域，最好再有一个二级子领域。

推荐流程：

```text
用户没有具体方向
→ 选择一级领域
→ 可选：选择二级子领域
→ M0 扫描趋势、空白、Web Search 信号
→ 生成多个候选课题
→ 用户选择一个候选课题
→ M1 问题精炼
→ 后续主流程继续
```

一级领域示例：

```text
1. 计算机科学 / AI
2. 医学 / 医疗 AI
3. 生物信息学
```

如果选择“计算机科学 / AI”，可以继续选择二级子领域：

```text
1. NLP / LLM
2. RAG / GraphRAG
3. Agent / Multi-Agent
4. 计算机视觉
5. 软件工程智能体
6. 推荐系统
7. 安全与隐私
8. AI Infra / MLOps
```

如果选择“医学 / 医疗 AI”，可以继续选择二级子领域：

```text
1. 医学问答
2. 医学影像
3. 临床预测
4. 医学知识图谱
5. 药物发现
6. 医疗 Agent
7. 电子病历分析
```

如果选择“生物信息学 / 计算生物学”，可以继续选择二级子领域：

```text
1. 基因表达分析
2. 单细胞组学
3. 蛋白质功能预测
4. 药物靶点发现
5. 多组学融合
6. 生物医学知识图谱
7. 生物序列建模
```

当前版本暂不覆盖材料科学、金融、教育、法律等其他领域。这些可以作为后续扩展方向，但不进入第一阶段产品范围。

M0 可以输出候选列表，例如：

```text
候选 1：面向医学问答的置信度感知 GraphRAG 幻觉抑制方法
候选 2：面向长文档问答的多跳证据链 RAG 评估框架
候选 3：面向科研 Agent 的失败记忆驱动 Prompt 自进化方法
```

用户选择后，系统再把该候选方向送入 M1 精炼成正式研究问题。

更完整的 CLI 分流可以是：

如果 y：
  请输入你的研究方向

如果 N：
  请选择一级领域

  "primary_domain": "计算机科学 / AI",

### 分流逻辑伪代码

```python
def start_research(
    user_has_direction: bool,
    raw_direction: str | None = None,
    primary_domain: str | None = None,
    subdomain: str | None = None,
):
    if user_has_direction and raw_direction:
        return run_pipeline(raw_question=raw_direction, skip_m0=True)

    if not primary_domain:
        primary_domain = ask_user_to_select_primary_domain()

    if subdomain is None:
        subdomain = ask_user_to_select_optional_subdomain(primary_domain)

    candidates = run_m0_discovery(
        primary_domain=primary_domain,
        subdomain=subdomain,
    )
    selected = ask_user_to_select(candidates)
    return run_pipeline(raw_question=selected.topic, skip_m0=True)
```

## M0 的目标

M0 的目标不是直接生成最终论文题目，而是生成一组“值得进一步验证的候选课题方向”。

每个候选方向应包含：

- 研究主题
- 背景动机
- 相关热点论文
- 潜在研究空白
- 可用数据集或 benchmark
- 可能 baseline
- 初步可行性评分
- 风险说明

示例输出：

```json
{
  "topic": "面向医学问答的置信度感知 GraphRAG 幻觉抑制方法",
  "motivation": "近两年医学 RAG、faithfulness evaluation、GraphRAG 相关工作增长明显，但实体链接误差和知识冲突处理仍不充分。",
  "evidence_papers": ["paper_id_1", "paper_id_2", "paper_id_3"],
  "research_gap": "现有方法多关注检索增强或知识图谱接入，但较少显式建模实体链接置信度对答案幻觉的影响。",
  "datasets": ["PubMedQA", "BioASQ", "MedQA"],
  "baselines": ["Vanilla RAG", "Self-RAG", "Corrective RAG", "KG-RAG"],
  "feasibility_score": 8.0,
  "risk": "医学领域评估复杂，需要谨慎处理事实性和临床安全性。"
}
```

## 模型路由与成本控制设想：GPT / Claude 中转站版

> 本节用于明确最终模型接入策略：后续新增架构不再采用 DeepSeek 优先方案，而是全部通过中转站统一接入 GPT 与 Claude。业务模块继续只依赖 `chat / reasoner / critical` 语义角色，不直接绑定具体厂商 SDK。

### 核心原则

```text
全部模型调用通过中转站
常规任务走 GPT
关键裁决走 Claude
业务模块只写 chat / reasoner / critical
模型替换、降级、成本统计集中在 LLM factory / RelayClient
```

### 推荐角色映射

```text
MODEL_CHAT      = gpt-5.5
MODEL_REASONER  = gpt-5.5
MODEL_CRITICAL  = claude-opus-4.7
```

> 具体模型名以中转站实际暴露名称为准。代码层建议保留 `MODEL_CHAT`、`MODEL_REASONER`、`MODEL_CRITICAL` 这类配置键，避免业务模块绑定具体模型版本。

### 各模块模型分工

| 模块 | 推荐角色 | 推荐模型路径 |
|---|---|---|
| M0 候选课题发现 | reasoner / critical | GPT via 中转站；最终 Top-K 裁决可用 Claude |
| M1 问题精炼 | reasoner | GPT via 中转站 |
| M2 query rewrite / 摘要处理 | chat | GPT via 中转站 |
| M3 三元组抽取 / GapCard 初稿 | chat / reasoner | GPT via 中转站 |
| M4 普通 Reviewer | reasoner | GPT via 中转站 |
| M4 Meta Reviewer | critical | Claude via 中转站 |
| M5 实验方案 | reasoner | GPT via 中转站 |
| M6 代码生成 | chat / reasoner | GPT via 中转站 |
| M7 写作 | chat | GPT via 中转站 |
| M8 compare / merge | reasoner / critical | GPT 初评；必要时 Claude 复核 |

### 成本控制策略

虽然全部通过 GPT / Claude 中转站，但仍要避免所有节点都调用最贵模型。推荐按角色分层：

```text
80%-90% 调用：MODEL_CHAT，用于抽取、总结、写作、格式化
10%-20% 调用：MODEL_REASONER，用于问题精炼、实验设计、普通 reviewer
1%-5% 调用：MODEL_CRITICAL，用于 Meta Reviewer、ResearchGate、M8 winner merge 等关键裁决
```

### critical 节点双阶段裁决

critical 节点不应默认无限制调用 Claude，而应采用双阶段：

```text
Stage 1：GPT reasoner 给出初裁
Stage 2：只有满足触发条件时，才调用 Claude critical 复核
```

触发条件：

```text
M4 reviewer 方差过大
final_rating 位于灰区，例如 5.5-7.0
存在 blocking issue 但 Meta 仍建议通过
M8 多分支评分接近，winner 不明显
医学/安全/高成本实验等高风险方向
用户显式要求 strong_review=true
```

这样可以保留 Claude 在关键裁决上的价值，同时避免每次都把所有节点升级到 expensive critical 模型。

### 中转站职责

中转站在新版策略中是统一模型入口，负责：

```text
统一 API base URL
统一鉴权和限流
统一 GPT / Claude 模型接入
统一成本统计
统一 fallback 和降级
支持模型 A/B 测试
屏蔽不同 provider SDK 差异
```

推荐调用路径：

```text
M1-M8 业务模块
→ get_llm(role)
→ LLM factory
→ RelayClient
→ GPT / Claude via 中转站
```

### 与现有设计方案需要同步的地方

后续所有新增架构说明中，凡是涉及模型路由，统一采用：

```text
chat      → GPT via 中转站
reasoner  → GPT 高推理模型 via 中转站
critical  → Claude Opus via 中转站
```

不再把 DeepSeek 作为新增架构的默认模型路径。DeepSeek 只保留为历史实现背景或可选兼容项，不作为当前目标方案的一部分。
## 模型路由与中转站接入设想

当前项目代码中 LLM 工厂大致按角色选择模型：

```text
chat      → 日常生成、抽取、写作
reasoner  → 推理、评审、实验设计
critical  → 关键节点、Meta Reviewer、最终裁决
```

原始实现中常规节点主要使用 DeepSeek，关键裁决节点使用 Claude。后续架构建议改为通过中转站统一接入模型：

```text
chat / reasoner → GPT-5.5 via 中转站
critical        → Claude Opus 4.7 via 中转站
```

### 为什么使用中转站

使用中转站的目标是统一模型接入层，而不是让业务模块直接绑定某个厂商 SDK。

好处：

- 统一 API base URL、鉴权和限流
- 方便在不改 M1-M7 业务代码的情况下替换模型
- 便于为不同角色配置不同模型
- 可以保留 `chat` / `reasoner` / `critical` 这种语义化模型角色
- 方便后续做模型 A/B、降级和成本控制

### 推荐角色映射

```text
MODEL_CHAT      = gpt-5.5
MODEL_REASONER  = gpt-5.5
MODEL_CRITICAL  = claude-opus-4.7
```

其中：

- `MODEL_CHAT`：用于 M2 query rewrite、M3 抽取、M7 写作等常规生成任务
- `MODEL_REASONER`：用于 M1 问题精炼、M4 普通 reviewer、M5 实验设计等推理任务
- `MODEL_CRITICAL`：用于 M4 Meta Reviewer、ResearchGate、关键候选课题裁决等高风险决策

### 推荐保留语义角色，不在业务代码写死模型名

业务模块不应直接写：

```python
get_llm("gpt-5.5")
```

而应继续写：

```python
get_llm("chat")
get_llm("reasoner")
get_llm("critical")
```

具体模型映射放在配置层：

```text
chat      → gpt-5.5
reasoner  → gpt-5.5
critical  → claude-opus-4.7
```

这样以后如果要改成其他模型，只需要改配置和 LLM factory，不需要改 M1-M7。

### 中转站 Provider 抽象

建议新增或改造 LLM client：

```text
llm/
├── base.py
├── relay.py          # 中转站统一客户端
├── factory.py        # 角色到模型的映射
└── adapters/         # 可选：不同 wire_api 的兼容层
```

`RelayClient` 负责：

```text
base_url
api_key
model
chat_json
chat
token usage
cost tracking
error handling
```

### 配置示例

```env
LLM_PROVIDER=relay
RELAY_BASE_URL=https://your-relay.example.com/v1
RELAY_API_KEY=sk-xxx

MODEL_CHAT=gpt-5.5
MODEL_REASONER=gpt-5.5
MODEL_CRITICAL=claude-opus-4.7
```

如果中转站使用 OpenAI-compatible 接口，可以统一走 OpenAI 风格请求；如果中转站区分 OpenAI / Anthropic wire protocol，则 `RelayClient` 需要根据模型或配置选择不同适配器。

### M4 中的使用方式

M4 普通 reviewer：

```text
novelty / methodology / statistics / reproducibility / devil
→ reasoner
→ gpt-5.5 via 中转站
```

M4 最终裁决：

```text
meta reviewer
→ critical
→ claude-opus-4.7 via 中转站
```

这保留了原来“普通评审用推理模型，最终裁决用更强 critical 模型”的架构思想。

### 注意事项

如果中转站暴露新模型名，例如 `gpt-5.5`，而本地 CLI 或工具层没有该模型 metadata，可能出现模型元数据警告。这个问题属于客户端模型 metadata 与中转站模型名不一致，不影响后端架构设计，但需要在实际运行时处理：

- 优先更新客户端模型 metadata
- 或让中转站提供兼容模型别名
- 或在配置层把业务模型名和中转站真实模型名分开

建议区分：

```text
logical_model = reasoner
provider_model = gpt-5.5
client_metadata_model = gpt-5.4 或已知兼容名
```

这样可以降低新模型刚发布时的兼容风险。

## M0 可以怎么发现方向

### 1. 研究趋势识别

基于近 2 年论文元数据和知识图谱，识别增长快的子领域。

可用信号：

- 新增论文数量
- 新增方法节点数量
- 新增数据集或 benchmark 数量
- 引用增长速度
- arXiv / Semantic Scholar / OpenAlex 多源共同出现频率
- 图中局部子图密度增长

判断逻辑示例：

```text
某个子领域近 2 年新增节点密度高
→ 说明方向活跃
→ 可作为候选热点方向
```

例如：

```text
GraphRAG、Agentic RAG、Self-RAG、RAG evaluation、faithfulness evaluation
```

如果这些概念在近两年论文中频繁新增并互相关联，就可以认为这是一个热门研究区域。

### 2. 研究空白识别

基于知识图谱识别“问题节点多次出现，但解决方案不足”的区域。

可用信号：

- 某个问题被多篇论文反复提到
- 很多方法都指向该问题，例如 `method --improves--> hallucination`
- 但缺少稳定、公认、可复现的解决方案
- 缺少统一 benchmark 或评价指标

判断逻辑示例：

```text
问题节点出现频率高
但有效解决方案节点少
→ 可能存在研究空白
```

例如：

```text
hallucination
faithfulness
domain shift
long-context retrieval
multi-hop evidence grounding
```

这些节点如果被大量方法尝试改进，但仍然反复作为问题出现，就可能是候选研究空白。

### 3. GraphRAG 增强检索

M0 不应只依赖普通向量检索，还可以结合图路径检索。

普通 RAG：

```text
query → 向量检索 → 相似论文摘要
```

GraphRAG：

```text
query → 向量检索相关论文
      → 识别相关实体
      → 沿知识图谱扩展邻居 / 路径 / 子图
      → 返回论文证据 + 实体关系路径
```

这样可以得到更丰富的上下文：

```text
KG-RAG --uses--> UMLS
KG-RAG --improves--> factual consistency
KG-RAG --evaluates_on--> PubMedQA
KG-RAG --compares_with--> Vanilla RAG
```

这比单纯返回几段摘要更适合科研选题，因为它能体现：

- 方法之间的关系
- 常用数据集
- 常见 baseline
- 高频问题
- 潜在实验切入点

### 4. Web Search 工具层

M0 还可以新增一个 **Web Search 工具层**，用于补充论文数据库之外的实时信息。

论文数据库适合回答：

```text
这个方向有哪些正式论文？
有哪些 DOI / arXiv / venue / citation 信息？
```

Web Search 更适合回答：

```text
这个方向最近有没有新开源项目？
是否有 benchmark leaderboard？
是否有 GitHub repo / blog / tutorial / demo？
是否有最新模型、数据集、工具链发布？
是否有社区正在讨论的痛点？
```

因此，M0 可以把信息源分成两层：

```text
学术检索层：arXiv / Semantic Scholar / OpenAlex / Crossref
Web Search 层：搜索引擎 / GitHub / Papers with Code / 官方博客 / leaderboard / 技术报告
```

Web Search 的价值是补齐“论文还没正式发表，但社区已经很活跃”的信号。例如：

- 新 benchmark 发布在 GitHub 或官方网页，论文还没被数据库收录
- 新模型先发 blog / technical report，正式论文滞后
- 某个方向 GitHub star 增长快，说明工程社区关注度高
- leaderboard 上出现新任务或新指标，说明评测体系正在形成
- 产业界正在使用某类方法，说明课题有现实价值

候选方向评分时可以新增 `web_signal_score`：

```text
web_signal_score = GitHub 活跃度
                 + leaderboard 是否存在
                 + 官方发布/技术博客数量
                 + 近期开源项目数量
                 + 社区讨论热度
```

需要注意的是，Web Search 信息噪声更高，应该遵循：

- 优先使用官方来源、论文主页、GitHub、Papers with Code、机构博客
- 将网页信息作为辅助信号，不直接替代论文证据
- 对网页来源记录 URL、访问时间、摘要和可信度
- 对非论文来源打上 `source_type=web`，避免和正式论文混淆

建议输出结构：

```json
{
  "query": "GraphRAG benchmark 2025",
  "title": "Example GraphRAG Benchmark",
  "url": "https://example.com/graphrag-benchmark",
  "source_type": "web",
  "credibility": "official_project",
  "summary": "该页面提供 GraphRAG 多跳问答 benchmark 和 leaderboard。",
  "signals": ["benchmark", "leaderboard", "open_source"]
}
```

## 建议新增的数据结构

可以在 `ResearchState` 中增加候选课题字段：

```python
class CandidateTopic(TypedDict, total=False):
    topic: str
    motivation: str
    evidence_papers: list[str]
    research_gap: str
    trend_score: float
    novelty_score: float
    feasibility_score: float
    web_signal_score: float
    datasets: list[str]
    baselines: list[str]
    web_evidence: list[dict[str, str]]
    risks: list[str]

class ResearchState(TypedDict, total=False):
    candidate_topics: list[CandidateTopic]
    selected_topic: CandidateTopic
```

同时，为了让 M3 的“研究空白识别”不再只是简单字符串列表，建议新增结构化的实体、关系和 Gap 证据卡片。

当前 M3 的简化输出类似：

```python
triples: list[Triple]
research_gaps: list[str]
```

建议升级为：

```python
class Entity(TypedDict, total=False):
    name: str
    type: str  # method / task / dataset / metric / problem / resource / paper / unknown
    aliases: list[str]
    evidence_papers: list[str]
    first_seen_year: int
    last_seen_year: int
    frequency: int

class Relation(TypedDict, total=False):
    head: str
    relation: str
    tail: str
    source_paper_id: str
    year: int
    confidence: float

class GapCard(TypedDict, total=False):
    gap: str
    problem_entities: list[str]
    evidence_papers: list[str]
    recentness: str
    existing_solutions: list[str]
    why_unsolved: str
    datasets: list[str]
    baselines: list[str]
    metrics: list[str]
    feasibility_score: float
    novelty_score: float
    evidence_score: float
    risk: str

class ResearchState(TypedDict, total=False):
    entities: list[Entity]
    relations: list[Relation]
    gap_cards: list[GapCard]
```

其中：

- `entities` 用来区分方法、任务、数据集、指标、问题等不同类型节点
- `relations` 用来保存带年份、置信度和来源论文的关系
- `gap_cards` 用来表达“为什么这是一个研究空白”，并附带证据、可行性和风险

保留旧的 `triples` 和 `research_gaps` 可以兼容现有流程，但后续 M4/M5 应优先读取 `gap_cards`。

## M3 增强：从简单启发式到 GapCard

当前 M3 的 `identify_research_gaps` 逻辑较轻量，主要根据图结构判断：

```text
某个节点被多次 improves / proposes 指向
并且这个节点自己很少作为 head 发出边
→ 可能是 research gap
```

这个逻辑适合作为教学版起点，但容易误判。例如：

```text
Method A --improves--> accuracy
Method B --improves--> accuracy
```

这里 `accuracy` 是指标，不是研究空白。又如：

```text
Method A --evaluates_on--> PubMedQA
```

`PubMedQA` 是数据集，也不是 gap。

因此，M3 需要增加实体类型和更丰富的关系类型。

### 实体类型识别

建议识别以下实体类型：

```text
method      方法，例如 KG-RAG / Self-RAG
task        任务，例如 medical QA / entity linking
dataset     数据集，例如 PubMedQA / BioASQ
metric      指标，例如 F1 / faithfulness / factuality
problem     问题，例如 hallucination / domain shift
resource    资源，例如 UMLS / knowledge graph
paper       论文
unknown     未知类型
```

只有 `problem` 类型节点才应该优先进入 gap 识别候选。

### 关系类型扩展

当前关系类型：

```text
improves
uses
compares_with
cites
proposes
evaluates_on
```

建议扩展：

```text
suffers_from
limited_by
fails_on
lacks
requires
future_work
open_problem
addresses
solves
```

这些关系能表达更接近科研空白的信息：

```text
Existing RAG --suffers_from--> hallucination
KG-RAG --limited_by--> entity linking error
Medical QA --lacks--> faithful citation evaluation
Paper X --future_work--> robust evidence grounding
Method Y --addresses--> hallucination
```

### 更稳的 gap 判断逻辑

新的 gap 判断不应只看节点计数，而应综合：

- 是否为 `problem` 类型实体
- 是否在近两年仍频繁出现
- 是否有 `limited_by` / `suffers_from` / `lacks` / `future_work` 证据
- 是否存在已有解决方案
- 现有解决方案是否不足
- 是否有公开数据集、baseline、metric
- 证据论文数量和质量是否足够
- Web Search 是否有 benchmark、GitHub、leaderboard 等实时信号

伪代码：

```python
for problem in problem_entities:
    evidence = collect_problem_evidence(problem)
    solutions = collect_solution_edges(problem)
    datasets = collect_related_datasets(problem)
    baselines = collect_related_baselines(problem)
    metrics = collect_related_metrics(problem)

    novelty = score_unsolvedness(problem, evidence, solutions)
    feasibility = score_feasibility(datasets, baselines, metrics)
    recentness = score_recentness(evidence.years)
    evidence_score = score_evidence_strength(evidence)

    if novelty > threshold and feasibility > threshold:
        gap_cards.append(build_gap_card(...))
```

### GapCard 示例

最终输出不应只是：

```text
["hallucination", "domain shift"]
```

而应是：

```json
{
  "gap": "医学 RAG 中实体链接错误导致的幻觉问题",
  "problem_entities": ["hallucination", "entity linking error", "medical QA"],
  "evidence_papers": ["paper1", "paper2", "paper3"],
  "recentness": "2023-2025 持续出现",
  "existing_solutions": ["KG-RAG", "Self-RAG"],
  "why_unsolved": "现有方法多假设知识图谱链接正确，较少建模实体链接置信度。",
  "datasets": ["PubMedQA", "BioASQ"],
  "baselines": ["Vanilla RAG", "KG-RAG", "Self-RAG"],
  "metrics": ["accuracy", "faithfulness", "citation precision"],
  "feasibility_score": 8.0,
  "novelty_score": 7.5,
  "evidence_score": 8.0,
  "risk": "医学事实性评估难度高，实体链接错误可能影响结论。"
}
```

## M2 增强：为 M3 提供更强证据

如果 M3 要更准确地识别 gap，M2 也应逐步增强开放获取和全文相关字段。

需要注意：系统不应把“必须拿到全文”作为运行前提。更合理的原则是：

```text
摘要可用于初筛；全文用于确认。
```

也就是说：

```text
M0 / M2 / M3：允许只基于标题、摘要、元数据做候选方向初筛
M4 / M5 / M7：尽量使用全文；如果缺全文，必须提示证据不足和风险
```

没有全文时，系统仍然可以继续工作，但不能假装已经读过全文。

建议在 `Paper` 中增加：

```python
pdf_url: str
is_open_access: bool
access_status: str  # open_access / metadata_only / restricted / unknown
full_text_available: bool
source_type: str     # paper / preprint / web / benchmark / repo
evidence_level: str  # fulltext / abstract_only / metadata_only / restricted
```

优先寻找合法开放来源：

```text
arXiv PDF
Semantic Scholar openAccessPdf
OpenAlex open_access / primary_location
PubMed Central
ACL Anthology
作者主页 PDF
```

第一版可以先不做全文解析，但应把字段和接口预留出来。后续可以抽取：

```text
Introduction
Related Work
Limitations
Future Work
Experiments
Datasets
Baselines
Metrics
```

这些章节比摘要更适合识别真实研究空白。

### 文献访问状态分层

建议把论文访问状态显式建模：

```text
open_fulltext      找到合法开放全文
abstract_only      只有摘要和元数据
metadata_only      只有标题、DOI、venue 等基础元数据
restricted         需要订阅、机构权限或用户提供 PDF
user_provided_pdf  用户手动提供 PDF
unknown            暂时无法判断
```

不同状态对应不同分析深度：

| 状态 | 可做分析 | 证据强度 |
|---|---|---|
| `open_fulltext` | 全文、limitations、future work、实验细节 | 强 |
| `user_provided_pdf` | 全文解析和深度分析 | 强 |
| `abstract_only` | 方向初筛、方法/任务/问题初步抽取 | 中 |
| `metadata_only` | 相关性和趋势弱参考 | 弱 |
| `restricted` | 记录元数据，提示用户可上传 PDF | 弱 |

可以在 M2 后增加一个轻量访问解析层：

```text
M2 检索论文元数据
→ M2.5 Access Resolver
→ 尝试寻找合法开放全文
→ 标记 access_status / evidence_level
→ M3 按证据等级处理
```

M2.5 可以依次尝试：

```text
arXiv PDF
Semantic Scholar openAccessPdf
OpenAlex open_access / primary_location
PubMed Central
ACL Anthology
作者主页 PDF
用户本地 PDF 目录匹配 DOI / title
```

如果都找不到，则不绕过权限，只标记为：

```text
abstract_only / metadata_only / restricted
```

### 证据等级降权

M3 / GapCard 生成时应根据证据等级降权，避免摘要级证据导致过度自信。

建议权重：

```text
全文证据：1.0
用户提供 PDF：1.0
摘要证据：0.5
元数据证据：0.2
受限文献：0.2
Web Search 辅助信号：0.3
```

例如某个 gap 的证据主要来自摘要，则可以继续保留，但置信度应降低：

```json
{
  "gap": "医学 RAG 中实体链接错误导致的幻觉问题",
  "evidence_level": "mixed",
  "fulltext_papers": 2,
  "abstract_only_papers": 6,
  "restricted_papers": 2,
  "confidence": 0.68,
  "note": "该 gap 主要基于摘要和部分全文推断，需要补充关键论文全文确认。"
}
```

### 快速模式与深度模式

可以把系统分成两种运行模式：

```text
快速模式：标题 + 摘要 + 元数据，适合 M0 候选课题发现和 M2/M3 初筛
深度模式：开放全文 + limitations + experiments + related work，适合 M4/M5/M7 深度确认
```

快速模式优势：

- 成本低
- 速度快
- 不依赖所有论文都有全文
- 适合生成多个候选方向

深度模式优势：

- 能确认 novelty
- 能检查 baseline 和实验设置
- 能抽取 limitations / future work
- 更适合最终课题确认和论文写作

最终原则：

```text
没有全文不是不能做，但必须标记不确定性；
摘要可以用于候选课题发现，最终确认和实验设计最好补全文。
```

## M4/M5 与 GapCard 的关系

增强 M3 后，M4/M5 不应只读取 `papers` 和 `research_gaps`，而应优先读取 `gap_cards`。

### M4：基于 GapCard 批判

M4 的 Reviewer 输入可以增加：

```text
候选 gap：医学 RAG 中实体链接错误导致幻觉
证据论文：paper1, paper2, paper3
已有方案：KG-RAG, Self-RAG
为什么未解决：多数方法假设实体链接正确，缺少置信度建模
可用数据集：PubMedQA, BioASQ
风险：医学事实性评估复杂
```

这样 Reviewer 可以更有针对性地判断：

- 这个 gap 是否真实
- 是否已经被已有方法解决
- 新颖性是否足够
- 实验是否能证明 claim
- 数据集和 baseline 是否合理

### M5：基于 GapCard 设计实验

M5 可以把 `selected_gap_card` 作为核心输入。

实验方案中的数据集、baseline、metric 不再完全依赖 LLM 临场生成，而是从 GapCard 中继承：

```text
datasets
baselines
metrics
existing_solutions
risks
```

这样 M5 生成的实验方案会更稳定，也更符合真实论文实验设计。

## 评分维度

M0 生成候选课题后，可以从以下维度评分：

| 维度 | 含义 |
|---|---|
| `trend_score` | 近两年热度是否上升 |
| `novelty_score` | 是否存在未充分解决的 gap |
| `feasibility_score` | 数据集、baseline、指标是否明确 |
| `evidence_score` | 是否有足够论文证据支撑 |
| `web_signal_score` | 是否有 GitHub、leaderboard、官方发布等实时信号 |
| `risk_score` | 伦理、安全、成本、复现风险 |

最终可以排序：

```text
final_score = 0.25 * trend_score
            + 0.30 * novelty_score
            + 0.25 * feasibility_score
            + 0.20 * evidence_score
            + 0.10 * web_signal_score
            - 0.20 * risk_score
```

## 分阶段实现建议

### Phase 1：轻量 M0

先不做复杂图数据库，只复用现有 M2/M3。

```text
输入宽泛领域
→ M2 检索近 2 年论文
→ M3 抽三元组
→ 统计高频问题 / 方法 / 数据集
→ LLM 生成 3-5 个候选方向
```

### Phase 2：趋势与空白分析

加入年份统计和图结构分析。

```text
按年份统计新增节点
识别高增长子图
识别高频问题节点
输出 trend_score / gap_score
```

### Phase 3：GraphRAG 检索

加入图路径检索。

```text
向量召回论文
实体链接到图谱节点
扩展 1-2 跳邻居
返回证据论文 + 图路径
```

### Phase 3.5：Web Search 工具层

加入实时 Web Search 信号，补充论文数据库之外的信息。

```text
候选方向关键词
→ Web Search 查询官方页面 / GitHub / leaderboard / 技术博客
→ 抽取 repo、benchmark、dataset、demo、release 信号
→ 生成 web_signal_score
→ 与论文证据一起进入候选方向评分
```

### Phase 4：自动进入主流程

让 M0 的最高分候选方向自动进入 M1-M5。

```text
M0 自动发现 5 个候选方向
→ M4 快速批判筛选
→ 选择最佳方向
→ M1 精炼成正式研究问题
→ 后续主流程继续运行
```

M0 的价值是让 AI Co-Scientist 从“被动科研助理”升级为“主动选题发现助手”。

## 架构优化方向与优先级

整体来看，本项目当前已经具备清晰的科研工作流：

```text
M1 定义问题
M2 找文献
M3 结构化知识
M4 批判
M5 定实验
M6 生成代码验证
M7 写论文
附录：记忆 / Prompt A/B / 技能库 / 对抗数据
```

它的优势是模块边界清楚、可解释性强、适合逐步增强。后续优化的核心方向是：

```text
从“LLM 根据摘要推理”升级为“基于证据卡片的科研工作流”。
```

### 1. M0 定位为候选课题发现器

M0 不应承诺“自动发现最终课题”，而应定位为：

```text
候选课题发现 + 证据卡片生成
```

每个候选课题都应带上：

- 相关论文
- 趋势证据
- gap 证据
- 数据集
- baseline
- Web Search 信号
- 风险
- 置信度

这样可以避免 M0 看起来像“AI 拍脑袋选题”。

### 2. M3 升级为 GapCard 生成层

这是最重要的架构升级之一。

当前 M3 更像：

```text
论文摘要 → 三元组 → 简单 research_gaps
```

建议升级为：

```text
论文证据 → 实体/关系/类型/时间/证据强度 → GapCard
```

也就是让 M3 输出：

```python
gap_cards: list[GapCard]
```

后续 M4/M5 都围绕 `GapCard` 工作。这样系统不只是发现一个“问题词”，而是生成一个有证据、有风险、有可行性判断的研究空白卡片。

### 3. M2.5 文献访问状态层

现实中很多论文没有全文权限，所以系统必须明确区分：

```text
open_fulltext
abstract_only
metadata_only
restricted
user_provided_pdf
```

建议架构：

```text
M2 检索论文
→ M2.5 Access Resolver
→ 标记 evidence_level
→ M3 按证据等级加权
```

这样系统可以继续使用摘要做初筛，但不会把摘要级分析伪装成全文分析。

### 4. M4 升级为 Evidence-grounded Reviewer

当前 M4 的 reviewer 主要是 prompt-based agent：

```text
novelty prompt
methodology prompt
statistics prompt
reproducibility prompt
devil prompt
meta prompt
```

这适合作为第一版，但后续应让 M4 读取更多结构化证据：

- `GapCard`
- 证据论文
- 访问状态
- baseline 列表
- dataset 列表
- Web Search 证据
- 受限文献风险

目标是把 M4 从：

```text
LLM 根据问题做泛泛评价
```

升级为：

```text
基于证据卡片做审稿式批判
```

当前 M4 的总体架构是合理的，尤其适合作为教学型/原型型多 Agent 审稿系统。它的主线是：

```text
Orchestrator 选择 Reviewer
→ 多个 Prompt-based Reviewer 并行审稿
→ 评分方差过大时触发 Devil's Advocate 二次批判
→ Meta Reviewer 汇总裁决
```

这个设计比“一个 LLM 直接评价方案”更强，因为它引入了多视角、结构化评分、分歧检测、魔鬼代言人和最终裁决。

#### M4 Agent 结构更新：Orchestrator-Subagent 动态评审团

结合 Phase 2 已实现的 `Orchestrator-Subagent` 动态选 Reviewer，新的 M4 不应再理解为“固定 5 个 reviewer 每次全员上场”，而应理解为一个分层 Agent 结构：

```text
M4 Evidence-grounded Roundtable
├─ Orchestrator Agent
│  ├─ 读取 refined_question / GapCard / evidence summary / method summary
│  ├─ 判断本次研究最需要哪些评审视角
│  └─ 动态选择 3-5 个 Reviewer Subagents
│
├─ Reviewer Subagents
│  ├─ novelty_reviewer
│  ├─ methodology_reviewer
│  ├─ statistics_reviewer
│  ├─ reproducibility_reviewer
│  ├─ devil_reviewer          # 强制保留，保证批判性
│  └─ domain_specific_reviewer # 后续按领域动态加入
│
├─ Debate / Recheck Layer      # 后续增强，可选
│  ├─ 当评分分歧大时触发二轮讨论
│  ├─ 当 blocking issue 出现时触发复审
│  └─ 当证据不足时要求补证据
│
└─ Meta Reviewer Agent
   ├─ 汇总所有 CritiqueCard
   ├─ 结合 reviewer_stats / evidence_level
   ├─ 输出 MetaDecision
   └─ 生成可被 M8 消费的 DecisionCard
```

这意味着 M4 的职责被拆成四层：

| 层级 | 角色 | 职责 |
|---|---|---|
| Orchestrator | 评审组长 | 决定本轮需要哪些 Reviewer，不直接做最终裁决 |
| Reviewer Subagents | 独立评审员 | 从不同角度输出结构化 `CritiqueCard` |
| Debate / Recheck | 二轮复审层 | 在分歧大、证据弱、blocking issue 多时触发 |
| Meta Reviewer | 最终裁判 | 汇总评审意见，输出最终决策与下一步动作 |

当前 Phase 2 已经完成的是前两层的核心能力：

```text
Orchestrator 动态选择 Reviewer
→ 被选中的 Reviewer 并行输出 CritiqueCard
→ Meta Reviewer 汇总裁决
```

后续架构升级的重点不是推翻现有 M4，而是把 M4 从“动态 reviewer 选择”继续升级为“证据驱动的评审委员会”：

```text
当前：refined_question + method_summary → Orchestrator 选 reviewer
升级：GapCard + evidence summary + access_status + experiment_brief → Orchestrator 选 reviewer
```

也就是说，Orchestrator 不只是根据问题文本选人，而是根据证据结构选人：

```text
如果 evidence_level 低 → 强制加入 reproducibility / methodology
如果涉及统计显著性 → 强制加入 statistics
如果涉及医学/临床 → 加入 clinical_safety_reviewer
如果涉及 RAG/Agent → 加入 retrieval_reviewer / agent_system_reviewer / evaluation_reviewer
如果评分分歧大 → 触发 Debate / Recheck Layer
```

#### M4 与 M8 的接口更新

为了让 M4 审核不通过后能驱动 M8 回退、分叉、重跑，M4 最终不应只输出 `MetaDecision`，还应输出面向流程控制的 `DecisionCard`。

推荐关系：

```text
CritiqueCard[]
→ Meta Reviewer 汇总
→ MetaDecision
→ DecisionCard
→ M8 replay / fork / stop / continue
```

推荐结构：

```python
class DecisionCard(TypedDict, total=False):
    passed: bool
    decision: str              # pass / minor_revision / major_revision / reject / stop
    final_rating: float
    recommended_action: str    # continue / refine_question / fetch_more_evidence / rebuild_gap / revise_experiment / stop
    target_node: str           # m1 / m2 / m3 / m5 / m6 / m7 / end
    branch_count: int
    branch_variants: list[str]
    blocking_issues: list[str]
    required_fixes: list[str]
    reason: str
```

这样 M4 和 M8 的边界会更清楚：

```text
M4 负责判断“这条路线哪里不行、下一步应该去哪”
M8 负责执行“回到哪个 checkpoint、开几条分支、怎么继续跑”
```

典型例子：

```json
{
  "passed": false,
  "decision": "major_revision",
  "final_rating": 5.8,
  "recommended_action": "fetch_more_evidence",
  "target_node": "m2",
  "branch_count": 3,
  "branch_variants": [
    "扩大检索范围，加入近两年论文",
    "只检索顶会和高引用论文",
    "加入代码仓库和 benchmark 检索"
  ],
  "blocking_issues": ["关键 baseline 证据不足"],
  "required_fixes": ["补充 Self-RAG / KG-RAG baseline 证据"],
  "reason": "当前证据不足以支持该研究 gap，建议回到 M2 补证据后重审。"
}
```

M8 读取后执行：

```text
target_node = m2
branch_count = 3
→ 从 M2 checkpoint 创建 3 条修正分支
→ 每条分支注入不同 branch_variant
→ 重新运行 M2 → M3 → M4
```

#### 新旧 M4 Agent 结构对比

| 版本 | Agent 结构 | 特点 |
|---|---|---|
| 原始版 | 固定 Reviewer 并行 + Meta | 简单稳定，但每次全员上场，相关性不强 |
| Phase 2 | Orchestrator 动态选 Reviewer + Meta | 根据问题动态组队，减少无关评审，提高信号密度 |
| 新架构目标 | Evidence-grounded Orchestrator + Reviewer + Debate + Meta + DecisionCard | 基于证据卡片审稿，并能驱动 M8 回退分叉 |

因此，新架构中的 M4 应被定位为：

```text
M4 = 动态评审团 + 证据审查层 + 流程决策器
```

但需要注意：M4 只输出决策，不直接执行分叉；真正执行分叉、回放、合并的是 M8。

#### M4 当前值得保留的设计

M4 现有设计中值得保留的部分：

- `novelty` / `methodology` / `statistics` / `reproducibility` / `devil` 分工明确
- `ReviewerPersona + system_prompt + model_role` 的实现简单、可扩展
- 多 Reviewer 并行审稿，降低耗时
- 方差过大时触发 Devil's Advocate 二次批判
- Meta Reviewer 不参与普通并行评审，而是在最后汇总裁决
- Reviewer 输出结构化 `CritiqueCard`，便于后续统计和展示

这些设计应继续保留，因为它们构成了 M4 的核心价值。

#### M4 当前主要不足

当前 M4 仍然更接近：

```text
Prompt-based simulated review panel
```

也就是“提示词驱动的模拟审稿会”。主要不足是：

1. **证据不够硬**

   当前 M4 主要读取 `refined_question`、`method_summary`、`experiment_brief` 和 `top_papers`。这些信息不足以支持严谨审稿。

   后续应增加：

   - `GapCard`
   - 证据论文表
   - 论文访问状态
   - 全文/摘要证据等级
   - baseline 证据
   - dataset 证据
   - Web Search 信号
   - 受限文献风险

2. **Reviewer 之间没有真正辩论**

   当前更像：

   ```text
   多人独立打分 → Meta 汇总
   ```

   Devil 二次批判是一个好的开始，但还不是完整 debate。

3. **缺少领域专家 Reviewer**

   当前 reviewer 是通用科研视角。对于医学 AI、材料科学、生物信息学等方向，最好动态加入领域专家：

   ```text
   domain_expert
   clinical_safety_reviewer
   bioinformatics_reviewer
   engineering_feasibility_reviewer
   ethics_reviewer
   ```

4. **评分标准需要校准**

   当前 reviewer 会输出 `rating 1-10`，但不同 reviewer 对分数的理解可能不一致。

   应增加统一 rubric。

5. **Meta Reviewer 需要结构化统计摘要**

   Meta 不应只阅读自然语言 card，也应看到：

   - `avg_rating`
   - `min_rating`
   - `max_rating`
   - `variance`
   - `lowest_reviewer`
   - `critical_risks`
   - `blocking_issues`

#### 建议的评分 Rubric

建议在 M4 prompt 中加入统一评分标准：

```text
9-10：强 accept，问题重要、创新明确、实验设计充分、风险低
7-8 ：有价值，但存在中等风险或需要补充实验
5-6 ：边界方案，需要 major revision，关键证据或实验不足
3-4 ：核心假设较弱，创新性或可行性明显不足
1-2 ：方向不成立、证据严重不足或实验不可行
```

这样能减少 reviewer 之间的评分漂移。

#### 建议增加 Debate Round

M4 可以从单轮评审升级为多轮审稿：

```text
Round 1：Reviewer 独立审稿
Round 2：每个 Reviewer 阅读其他人的主要批评后修正观点
Round 3：如果分歧仍大，Devil's Advocate 汇总最强反对意见
Round 4：Meta Reviewer 裁决
```

更具体地说，可以把“第一轮各自评分完以后进入对话”作为 M4 的核心升级：

```text
M4 Multi-round Review
│
├─ Round 1: Independent Review
│   ├─ novelty 独立评分
│   ├─ methodology 独立评分
│   ├─ statistics 独立评分
│   ├─ reproducibility 独立评分
│   └─ devil 独立找风险
│
├─ Round 2: Cross-review Discussion
│   ├─ 所有 Reviewer 阅读 Round 1 的 CritiqueCards
│   ├─ 各自说明同意/不同意哪些观点
│   ├─ 各自决定是否修正 rating
│   └─ 标记 blocking issues
│
├─ Round 3: Devil Synthesis, optional
│   └─ 若分歧仍大或出现 blocking issue，由 Devil 汇总最强反方观点
│
└─ Round 4: Meta Decision
    └─ Meta 读取全部 cards / discussion / devil synthesis 后裁决
```

这样 M4 不再只是：

```text
多个 Reviewer 各自写意见 → Meta 汇总
```

而是变成：

```text
独立审稿 → 交叉讨论 → 争议强化 → 最终裁决
```

这更接近真实审稿委员会的工作方式。

##### Round 1：独立审稿

每个 Reviewer 不看其他人的意见，独立输出 `CritiqueCard`：

```python
class CritiqueCard(TypedDict, total=False):
    reviewer: str
    soundness: int
    contribution: int
    presentation: int
    strengths: list[str]
    weaknesses: list[str]
    questions: list[str]
    limitations: list[str]
    rating: int
    confidence: int
    rationale: str
```

Round 1 的意义是保留每个 reviewer 的原始独立判断，避免过早群体共识导致偏见。

##### Round 2：交叉讨论

Round 2 让每个 reviewer 读取所有 Round 1 卡片，然后输出 `DiscussionCard`：

```python
class DiscussionCard(TypedDict, total=False):
    reviewer: str
    agreements: list[str]
    disagreements: list[str]
    revised_rating: float
    blocking_issues: list[str]
    changed_mind: bool
    rationale: str
```

Round 2 的 prompt 可以要求：

```text
你已经看到其他 Reviewer 的意见。
请回答：
1. 你同意哪些意见？
2. 你不同意哪些意见？
3. 你是否要修改自己的 rating？
4. 哪些问题是 blocking issue？
```

这样可以让不同 reviewer 互相校正。例如：

```text
Novelty Reviewer 发现已有相似工作
→ Methodology Reviewer 看到后下调评分
→ Reproducibility Reviewer 发现关键论文 restricted
→ Devil 强化“证据不足”风险
```

##### Round 3：Devil Synthesis

如果第二轮后仍然分歧大，或出现严重 blocking issue，则触发 Devil 综合反方意见。

Devil 读取：

```text
Round 1 CritiqueCards
Round 2 DiscussionCards
GapCard
Evidence summary
```

然后输出：

```python
class DevilSynthesis(TypedDict, total=False):
    strongest_objections: list[str]
    weakest_assumptions: list[str]
    must_fix_before_continue: list[str]
    fatal_risk: bool
    rationale: str
```

##### Round 4：Meta Decision

Meta Reviewer 最后读取：

```text
Round 1 CritiqueCards
Round 2 DiscussionCards
DevilSynthesis
Reviewer stats
Evidence summary
GapCard
```

然后输出最终 `MetaDecision`。

这个结构可以让 Meta 不只是“平均评分”，而是基于完整讨论过程裁决。

##### Round 2 触发策略

建议支持两种模式：

```text
快速模式：默认跳过 Round 2，除非方差大或风险高
深度模式：默认运行 Round 2
```

触发逻辑：

```python
if review_mode == "deep":
    run_discussion = True
elif variance > threshold:
    run_discussion = True
elif domain in HIGH_RISK_DOMAINS:
    run_discussion = True
elif has_blocking_issue:
    run_discussion = True
else:
    run_discussion = False
```

高风险领域可以包括：

```text
medical / clinical / finance / legal / safety / biosecurity
```

##### 对项目的价值

加入 Round 2 后，M4 从：

```text
并行打分 + Meta 总结
```

升级为：

```text
审稿委员会讨论 + 修正意见 + Meta 裁决
```

这会让 M4 的多 Agent 不只是“多提示词并行”，而是真正具备一定协商能力。

触发 Debate Round 的条件可以是：

```text
rating 方差 > 阈值
最低分 Reviewer 给出 blocking issue
Devil's Advocate rating 明显低于平均分
证据等级以 abstract_only 为主
关键文献 restricted 且缺全文
```

#### 建议增加领域专家 Reviewer

Orchestrator 应根据领域动态选择领域 reviewer。

例如医学方向：

```text
medical_domain_reviewer
clinical_safety_reviewer
```

RAG / Agent 方向：

```text
retrieval_reviewer
agent_system_reviewer
evaluation_reviewer
```

生物信息方向：

```text
bioinformatics_reviewer
data_quality_reviewer
```

这样可以避免通用 reviewer 忽略专业风险。

#### 建议增加规则兜底

Orchestrator 可以用 LLM 动态选择 reviewer，但也应有规则兜底。

例如只要输入中出现：

```text
dataset / metric / experiment / baseline / p-value / seed / ablation
```

就强制加入：

```text
statistics reviewer
methodology reviewer
reproducibility reviewer
```

如果出现：

```text
medical / clinical / patient / diagnosis / treatment
```

就强制加入：

```text
clinical_safety_reviewer
```

#### 建议的新 M4 输出结构

M4 最终输出不应只包含：

```text
decision / final_rating / verdict
```

还应增加可直接指导后续流程的字段：

```python
class MetaDecision(TypedDict, total=False):
    decision: str
    final_rating: float
    key_strengths: list[str]
    key_risks: list[str]
    verdict: str
    blocking_issues: list[str]
    required_fixes: list[str]
    missing_evidence: list[str]
    reviewer_stats: dict[str, float]
    recommended_next_step: str  # revise_experiment / fetch_fulltext / continue_to_m5 / reject
```

示例：

```json
{
  "decision": "major_revision",
  "final_rating": 6.5,
  "blocking_issues": [
    "关键 baseline 缺少全文或代码证据",
    "当前数据集不足以验证医学事实性 claim"
  ],
  "required_fixes": [
    "补充 Self-RAG 和 KG-RAG baseline",
    "增加 citation faithfulness 指标",
    "上传或获取关键受限论文 PDF"
  ],
  "missing_evidence": ["paper_id_3", "paper_id_7"],
  "recommended_next_step": "fetch_fulltext"
}
```

#### M4 升级后的目标形态

最终 M4 可以升级为：

```text
M4 Evidence-grounded Roundtable
```

流程：

```text
输入：
- refined_question
- selected_gap_card
- top_papers
- evidence_level summary
- experiment_brief

Step 1：Orchestrator 选择 reviewer
Step 2：Reviewer 独立审稿
Step 3：聚合 blocking issues
Step 4：如果分歧大或 blocking issue 多，触发 Debate Round
Step 5：Meta Reviewer 基于 cards + statistics + evidence summary 裁决
Step 6：输出 decision + required_fixes + recommended_next_step
```

一句话总结：

```text
M4 当前方向是对的，但现在更像“提示词驱动的模拟审稿会”；下一步应升级成“基于证据卡片的结构化审稿委员会”。
```

### 5. M5 继承证据而不是重新发明实验

M5 设计实验时不应完全依赖 LLM 临场生成数据集、baseline 和指标。

更合理的方式是：

```text
M5 = GapCard → ExperimentPlan
```

M5 应优先继承 `GapCard` 中的：

- `datasets`
- `baselines`
- `metrics`
- `existing_solutions`
- `risks`

LLM 负责组合、补全和解释，而不是凭空生成实验要素。

### 6. M6 执行验证分层

M6 是项目亮点，但真实科研实验通常不是 toy sample 能完全验证的。

更重要的是，本项目后续覆盖计算机、医疗、生物三个方向后，M6 不应固定理解为“代码生成模块”。更合理的抽象是：

```text
M6 = Verification Layer / 可验证产物生成层
```

不同领域的“可验证产物”不同：

| 领域 | M6 产物 | 说明 |
|---|---|---|
| 计算机科学 / AI | 代码原型、toy run、指标计算脚本 | 适合生成 Python 项目并运行最小样本 |
| 医疗 AI / 医学信息学 | 统计分析计划、实验 protocol、样本量估算、数据字段检查脚本 | 不应假装完成真实临床验证 |
| 生物信息学 / 计算生物学 | 数据处理 pipeline、公开数据集分析 notebook、统计检验脚本、候选基因/靶点筛选流程 | 适合做公开数据和计算分析验证 |

也就是说：

```text
不是所有课题都生成代码；
但所有课题都应该生成某种可验证产物。
```

建议将 M6 从：

```text
M6 Code Generation
```

升级为：

```text
M6 Verification Layer
```

注意：这不是要删除原始 M6。原始的 **“代码自动生成 + 沙箱执行（两段式 + 开关）”** 架构应当保留，并作为计算机科学 / AI 方向的默认 verification backend。

也就是说，新旧关系不是：

```text
旧 M6 被新 M6 替代
```

而是：

```text
旧 M6 被纳入新 M6
```

旧 M6 负责计算机 / AI 方向的代码原型验证，新 M6 负责跨领域的“可验证产物生成”抽象。

并在内部根据领域选择不同执行类型：

```python
execution_type = Literal[
    "code_prototype",              # 计算机 / AI
    "statistical_analysis_plan",   # 医疗 / 生物统计
    "experimental_protocol",       # 医疗 protocol
    "bioinformatics_pipeline",     # 生物信息分析流程
    "public_dataset_notebook",     # 公开数据集 notebook
    "no_execution",                # 理论型或暂不适合执行
]
```

### 三个领域下的 M6 设计

#### 计算机科学 / AI

适合继续保留当前 M6 的代码生成能力：

```text
生成代码骨架
跑 toy sample
计算指标
验证 baseline 流程
```

典型产物：

```text
main.py
dataset.py
model.py
metrics.py
README.md
requirements.txt
```

原始 M6 的两段式设计继续保留：

```text
M6-A：Code Generator
→ 根据 M5 实验方案生成代码文件、requirements、README

M6-B：Code Executor
→ 根据执行开关决定是否运行生成代码
```

执行开关继续保留：

```text
generate_only  只生成代码，不执行
dry_run        做轻量检查，例如语法、import、文件结构
full_execute   在 Docker 沙箱中运行 toy sample / 最小实验
```

这个设计的价值是：

- 代码生成和代码执行解耦，便于人工检查
- 用户可以控制执行风险
- 沙箱隔离生成代码，避免污染宿主环境
- `generate_only` 适合快速预览
- `dry_run` 适合低成本发现语法和依赖问题
- `full_execute` 适合验证最小实验流程是否跑通
- 成功执行后仍可进入 SkillLibrary，沉淀可复用函数

在新架构中可以表示为：

```text
M6 Verification Layer
├── code_generation_backend        # 计算机 / AI 默认后端，保留原始 M6
│   ├── M6-A Code Generator
│   ├── M6-B Code Executor
│   ├── generate_only
│   ├── dry_run
│   └── full_execute
├── medical_protocol_backend       # 医疗方向
└── bioinformatics_pipeline_backend # 生物信息方向
```

因此，原始 M6 仍然是项目亮点之一，只是它在多领域架构中变成了 `code_prototype` 类型的实现分支。

#### 医疗 AI / 医学信息学

医疗方向不应直接生成“临床实验结果”，而应生成合规的验证材料：

```text
统计分析计划
样本量估算
数据字段检查脚本
纳入/排除标准
风险与伦理检查清单
实验 protocol 草案
```

如果有公开医学数据集，可以生成分析脚本；如果涉及真实临床数据，则必须标记权限、伦理和隐私风险。

#### 生物信息学 / 计算生物学

生物方向可以生成计算分析 pipeline：

```text
公开数据集下载说明
表达矩阵预处理脚本
差异表达分析
富集分析
多组学融合流程
候选基因/靶点筛选 notebook
统计检验脚本
```

如果涉及湿实验验证，M6 只能生成 protocol 或计算预测结果，不能假装完成真实实验。

建议把执行验证分层：

```text
syntax_check   语法检查
toy_run        小样本跑通
unit_eval      最小指标验证
full_execute   完整实验或尽可能接近完整实验
```

当前的 `full_execute` 更接近 toy execution / sandbox execution，文档和 UI 中应避免让用户误解为完整科研实验已经被验证。

### M5.5 ResearchGate：通过/回退决策层

真实科研流程不是线性的。如果 M4/M5 判断课题不可靠，系统不能继续硬推到 M6/M7，而应进入回退循环。

建议在 M5 之后新增：

```text
M5.5 ResearchGate
```

它负责读取：

```text
GapCard
M4 MetaDecision
ExperimentPlan
EvidenceLevel
FeasibilityScore
领域类型
```

然后决定：

```text
继续进入 M6
回到 M5 修改实验方案
回到 M2/M2.5 补证据或全文
回到 M1 重写研究问题
回到 M0 重新选择课题
终止当前方向
```

ResearchGate 输出结构示例：

```python
class ResearchGateDecision(TypedDict, total=False):
    passed: bool
    reason: str
    recommended_action: str  # continue_to_m6 / revise_experiment / fetch_more_evidence / refine_question / choose_new_topic / stop
    target_node: str         # m6 / m5 / m2_5 / m2 / m1 / m0 / end
    fixes: list[str]
    execution_type: str
```

示例：

```json
{
  "passed": false,
  "reason": "关键 baseline 缺少全文和开源实现，当前实验方案不可验证。",
  "recommended_action": "fetch_more_evidence",
  "target_node": "m2_5",
  "fixes": [
    "寻找 KG-RAG 和 Self-RAG 的开放全文或代码仓库",
    "如果无法获取，则替换为可复现 baseline",
    "重新计算 feasibility_score"
  ],
  "execution_type": "code_prototype"
}
```

### 按失败原因回退

ResearchGate 可以根据失败原因选择不同回退节点：

| 失败原因 | 回退节点 | 动作 |
|---|---|---|
| 创新性不足 | M0 / M3 | 换候选 gap，扩大或缩小方向，补近两年论文 |
| 数据集不可用 | M5 | 更换数据集，缩小任务，改成 benchmark 构建型课题 |
| Baseline 不清楚 | M2 / M2.5 / M5 | 补全文、找代码、换 baseline |
| 方法不可行 | M5 | 降低方法复杂度，改成轻量实验或分析型课题 |
| 证据不足 | M2.5 | 找开放全文，请用户上传 PDF，降低证据置信度 |
| 领域风险过高 | M4 / M5 / END | 加领域专家审查，或终止当前方向 |

### 推荐状态机

整体状态机可以设计为：

```text
START
→ Direction Intake
→ M0 or M1
→ M2
→ M2.5 Access Resolver
→ M3 GapCard
→ M4 Evidence-grounded Roundtable
→ M5 Experiment Design
→ M5.5 ResearchGate
   ├─ pass → M6 Verification Layer
   ├─ revise_experiment → M5
   ├─ fetch_more_evidence → M2/M2.5
   ├─ refine_question → M1
   ├─ choose_new_topic → M0
   └─ stop → END
→ M7 Paper Draft
```

### M5 不通过时的处理原则

如果 M5 或 ResearchGate 判断“不通过”，不应直接失败，而应给出明确下一步：

```text
如果问题还值得做 → 回退修改
如果证据不足 → 补证据
如果方向不成立 → 回到 M0 换课题
如果风险不可接受 → 终止
```

这样系统不会把低质量课题硬推进到代码生成和论文生成阶段。

### 7. 记忆库、Prompt A/B、技能库增加质量门槛

进化记忆、Prompt A/B、技能库都很有价值，但需要避免低质量内容污染长期状态。

建议只有满足以下条件时才沉淀为长期记忆或技能：

```text
M4 final_rating >= 阈值
M6 验证成功
用户确认有价值
```

技能库还可以增加：

- 函数可复用性判断
- 使用次数统计
- 过期/淘汰机制
- 用户确认机制

这样可以避免“跑通过一次”的低质量函数长期污染技能库。

### 8. 增加用户确认节点

科研选题不适合完全无确认地自动推进。建议在关键节点加入人工确认：

```text
M0 后：用户选择候选课题
M3 后：用户确认 gap 是否合理
M4 后：用户确认是否继续
M5 后：用户确认实验方案
M6 前：用户确认是否执行代码
```

这既能提升安全性，也更符合真实科研工作流。

### 9. 前端重点展示证据链

如果继续做 Web 前端，重点不应只是聊天 UI，而应展示科研工作流中的证据链：

```text
候选课题卡片
文献证据列表
知识图谱
GapCard
Reviewer 批判卡片
实验方案表
代码验证状态
论文草稿
```

其中最有价值的可视化对象是：

- `GapCard`
- `Reviewer Cards`
- 文献访问状态
- 知识图谱子图
- 实验方案对照表

### 推荐实施优先级

建议按以下顺序推进：

```text
1. 先补 M0 交互分流和候选课题卡片
2. 再把 M3 research_gaps 升级成 GapCard
3. 加 M2.5 access_status / evidence_level
4. 让 M4/M5 读取 GapCard
5. 再考虑 GraphRAG 和 Web Search 深度增强
```

不要一开始就做完整 GraphRAG、全文解析和 Web Search，否则范围容易失控。第一阶段应追求“可用、可解释、可逐步增强”。

## 补充方案：有 M0 时的 M8 分支并行架构

如果实现 M0 自动选题发现模块，M8 的最佳用法是：让 M0 先产生多个候选研究方向，再由 M8 把这些方向变成多条 Git-like 研究分支并行探索。

一句话概括：

```text
M0 = 候选课题 / 候选路线生成器
M8 = 分支、回放、比较、合并管理器
M4 = 每条分支的质量审核器
```

### 推荐主流程

有 M0 版本推荐采用：

```text
用户输入研究领域 / 兴趣方向 / 约束条件
→ M0 自动发现多个 TopicCard / GapCard
→ M0 或用户选择 Top-K 候选方向
→ M8 为每个候选方向创建 fork
→ 每条 fork 独立运行 M1-M7
→ 分支内部遇到审核不通过时，M8 回退到对应节点重跑
→ M8 compare 横向比较所有分支
→ M8 merge 最优分支为主线
```

也可以画成：

```text
用户输入领域或粗粒度方向
  ↓
M0 自动选题发现
  ├─ Topic A / Gap A
  ├─ Topic B / Gap B
  └─ Topic C / Gap C
  ↓
M8 创建研究分支
  ├─ fork_A: Topic A → M1 → M2 → M3 → M4 → M5 → M6 → M7
  ├─ fork_B: Topic B → M1 → M2 → M3 → M4 → M5 → M6 → M7
  └─ fork_C: Topic C → M1 → M2 → M3 → M4 → M5 → M6 → M7
  ↓
M8 compare
  ↓
merge winner → 主线结果
```

### M0 输出什么给 M8

M0 不应该直接输出最终研究结论，而应该输出可分叉执行的候选卡片。

推荐结构：

```python
TopicCard:
    topic_id: str
    title: str
    research_direction: str
    candidate_question: str
    suspected_gap: str
    key_evidence: list[str]
    novelty_rationale: str
    feasibility_rationale: str
    risk_factors: list[str]
    score: float
```

M8 消费 `TopicCard` 时，每张卡片对应一个 fork：

```text
TopicCard A → fork_A.raw_question / fork_A.initial_context
TopicCard B → fork_B.raw_question / fork_B.initial_context
TopicCard C → fork_C.raw_question / fork_C.initial_context
```

### M8 从哪里开始跑

有 M0 时，M8 通常不需要从 M0 重新跑每条分支，而是从 M1 开始：

```text
M0 已经完成候选方向发现
→ 每条 fork 拿到一个 candidate_question
→ 从 M1 开始把候选方向精炼成正式 PICO / 研究问题
```

因此默认分支起点是：

```text
branch_start_node = m1
```

只有当某条分支被 M4 判定为“方向本身不成立”或“创新性不足”时，才需要回到 M0 换新的候选课题：

```text
M4: choose_new_topic
→ M8 回到 M0
→ 生成新的 TopicCard
→ 再从 M1 开新 fork
```

### 分支内部的回退逻辑

每条 fork 都是独立研究路线。分支内部如果失败，不一定整条废弃，而是由 M4/M5/M6 输出决策，再让 M8 回退到对应节点：

```text
创新性不足      → 回到 M0，换候选课题
问题定义不清    → 回到 M1，重写 PICO
证据不足        → 回到 M2，补充检索
gap 不成立      → 回到 M3，重构 GapCard
实验不可执行    → 回到 M5，重设实验方案
代码验证失败    → 回到 M6，换实现策略；或回到 M5，降级实验
论文表达不充分  → 回到 M7，重写草稿；或回到 M4/M5 补强证据
```

这时 M8 的职责是执行：

```text
找到目标节点 checkpoint
→ 基于该 checkpoint 创建新 fork
→ 注入修正策略 branch_variant
→ 从目标节点继续跑
```

### M4、M0、M8 的边界

三个模块边界建议明确写死，避免职责混乱：

```text
M0 负责提出候选方向，不负责长期管理分支。
M4 负责审核单条路线，不负责创建分支。
M8 负责分支、回放、比较、合并，不负责判断科学质量。
```

它们的连接方式是：

```text
M0 输出 TopicCard
M8 消费 TopicCard 创建 fork
M4 输出 DecisionCard
M8 消费 DecisionCard 执行 replay / fork / stop / merge
```

### 有 M0 版本的 merge 定义

第一阶段不要做复杂的自动融合，先做“选择最优分支成为主线”：

```text
merge = 选择评分最高 / 用户确认的 winner fork，并标记为 mainline
```

主线结果包括：

```text
winner TopicCard
winner PICO
winner evidence set
winner GapCard
winner critique cards
winner experiment plan
winner code result
winner paper draft
```

后续可以升级为“吸收其他分支优点”：

```text
merge = winner fork 主体 + 其他 fork 的补充证据 / reviewer 建议 / 实验设计优点
```

### 有 M0 版本的实施优先级

如果选择实现 M0，推荐实施顺序为：

```text
1. 定义 TopicCard / GapCard 数据结构
2. 实现 M0 Top-K 候选方向生成
3. 扩展 M8：支持 TopicCard 批量 create_fork
4. 扩展 graph/API：每个 fork 可携带不同 initial_topic / raw_question
5. 扩展 M4：输出 DecisionCard，明确 target_node 和 recommended_action
6. 扩展 M8 compare：展示每条 fork 的评分、证据、实验可行性、论文草稿
7. 实现轻量 merge：将 winner fork 标记为 mainline
8. 后续再做跨分支内容融合
```

### 有 M0 与无 M0 的区别

两种方案的核心区别是“候选路线从哪里来”：

| 方案 | 候选路线来源 | M8 默认分支起点 | 适合阶段 |
|---|---|---|---|
| 有 M0 | M0 自动发现 TopicCard / GapCard | M1 | 主动选题发现版本 |
| 无 M0 | M1 生成多个 PICO / 问题变体 | M2 或 M1 后续 | 更容易落地的 MVP |

因此，有 M0 时系统更像“主动科研选题助手”；无 M0 时系统更像“同一用户问题的多路线研究探索器”。

## 补充方案：无 M0 时的 M8 分支并行架构

如果暂时不实现 M0 自动选题发现模块，M8 仍然有独立价值。M0 不是 M8 的必要条件，M8 真正需要的是“多个候选路线”。这些候选路线可以由 M1、M2/M3、M4、M5，甚至用户手动产生。

### 推荐选择：M1 生成多个研究问题变体，再由 M8 开分支

无 M0 版本的第一阶段推荐采用：

```text
用户输入 raw_question
→ M1 生成多个 PICO / 研究问题变体
→ M8 为 Top-K 变体创建 fork
→ 每条 fork 独立运行 M2-M7
→ M4/M7 对每条路径评分
→ M8 compare 横向比较
→ M8 merge 最优分支为主线
```

也可以画成：

```text
用户原始问题
  ↓
M1 问题精炼，生成 3 个候选 PICO
  ↓
M8 创建分支
  ├─ fork_A: PICO A → M2 → M3 → M4 → M5 → M6 → M7
  ├─ fork_B: PICO B → M2 → M3 → M4 → M5 → M6 → M7
  └─ fork_C: PICO C → M2 → M3 → M4 → M5 → M6 → M7
  ↓
M8 compare
  ↓
merge winner → 主线结果
```

### M1 在这里承担的角色

没有 M0 时，M1 不只做单一路径的问题精炼，而是升级为“轻量候选路线生成器”。

例如用户输入：

```text
如何减少 RAG 幻觉？
```

M1 可以生成：

```text
A: 基于 citation grounding 减少 RAG 幻觉
B: 基于 query rewriting 减少 RAG 幻觉
C: 基于 multi-agent verification 减少 RAG 幻觉
```

M8 再把这 3 个方向变成 3 条研究分支。这样不需要先实现完整 M0，也能做出“多路线探索 + 对比择优”的项目亮点。

### 分支内部的回退机制

每条 fork 内部仍然可以使用 M8 做局部回退和重跑：

```text
M4 审核不通过 → 根据失败原因回到 M1 / M2 / M3
M5 实验不可执行 → 回到 M5 生成新实验方案
M6 代码失败 → 回到 M6 重写代码，或回到 M5 降级实验方案
M7 论文质量低 → 回到 M7 重写表达，或回到 M4/M5 补证据
```

因此，无 M0 版本仍然保留完整的 Git-like 研究管理能力：

```text
fork → replay → compare → merge
```

### M4 与 M8 的关系

M4 负责判断单条路线是否合格，M8 负责执行分叉、回退和合并。两者不是重复模块，而是上下游关系：

```text
M4 = 质量门禁 / 审核决策
M8 = 版本管理 / 分支执行
```

建议让 M4 输出结构化决策卡片：

```python
DecisionCard:
    passed: bool
    recommended_action: str  # continue / refine_question / fetch_more_evidence / revise_experiment / stop
    target_node: str         # m1 / m2 / m3 / m5 / m6 / m7 / end
    branch_count: int
    branch_variants: list[str]
    reason: str
```

M8 消费这个 `DecisionCard`：

```text
passed = true  → 当前 fork 继续向后运行
passed = false → 根据 target_node 回到对应 checkpoint，并创建修正分支
stop           → 标记当前 fork 为 abandoned
```

### 第一阶段落地范围

为了避免范围过大，第一阶段只做“选择最佳分支成为主线”，暂不做复杂内容融合：

```text
merge = 将评分最高 / 用户确认的 fork 标记为 mainline / winner
```

后续再升级为：

```text
merge = 吸收其他分支的优点、证据、实验设计和 reviewer 建议
```

### 无 M0 版本的实施优先级

如果选择暂缓 M0，推荐新的实施顺序为：

```text
1. 扩展 M1：输出 Top-K PICO / ResearchQuestion candidates
2. 扩展 M8：支持从 M1 candidates 批量 create_fork
3. 扩展 graph/API：允许每个 fork 携带不同 refined_question / pico
4. 扩展 M4/M7：沉淀每条 fork 的评分与摘要
5. 扩展 M8 compare：横向展示 fork 的评分、证据、实验方案、论文草稿路径
6. 实现轻量 merge：将 winner fork 标记为主线
```

这个方案可以作为 M0 之前的过渡版本，也可以作为项目 MVP 的核心亮点：不做自动选题，但先实现“同一用户问题的多路线科研探索”。
