# 新增架构设想整理版：从单线科研 Agent 到多路线主动科研系统

> 本文是对 `新增架构设想_M0自动选题发现.md` 的结构化整理版，用于后续更新正式设计文档。原始文档保留为脑暴记录；本文按“目标 → 总架构 → 模块升级 → 数据结构 → 流程控制 → 实施路线”的顺序重排。

---

## 1. 背景与升级目标

当前项目已经实现了 AI Co-Scientist 的基础闭环：

```text
M1 问题精炼
→ M2 多源文献检索
→ M3 知识图谱构建
→ M4 多 Reviewer 批判
→ M5 实验方案设计
→ M6 代码生成与验证
→ M7 论文草稿生成
```

在上述基础上，Phase 1-3 已完成：

| Phase | 已完成升级 | 代表能力 |
|---|---|---|
| Phase 1 | MCP 工具层标准化 | M2 外部检索工具协议化 |
| Phase 2 | Orchestrator-Subagent | M4 动态选择 Reviewer |
| Phase 3 | 生产级基础设施 | LangSmith、Extended Thinking、Budget Guard |

新的架构设想目标是把系统从“用户给定问题后的单线科研助手”升级为：

```text
主动发现候选方向
→ 多路线并行探索
→ 证据驱动审稿
→ 按失败原因回退
→ 比较并合并最优研究路径
```

一句话定位：

```text
从 single-path research agent 升级为 multi-branch research operating system。
```

## 2. 新架构总览

### 2.1 新增与升级后的模块定位

| 模块 | 新定位 | 说明 |
|---|---|---|
| M0 | 候选课题发现器 | 生成 TopicCard / GapCard，不承诺直接给最终课题 |
| M1 | 问题精炼与 PICO 生成 | 有 M0 时精炼候选方向；无 M0 时可生成多个问题变体 |
| M2 | 多源证据检索层 | 提供论文、元数据、访问状态和证据等级 |
| M2.5 | 文献访问状态层 | 判断 fulltext / abstract_only / restricted / code_available |
| M3 | GapCard 生成层 | 从知识图谱和证据中构造结构化研究空白 |
| M4 | Evidence-grounded Roundtable | 动态评审团 + 证据审查 + 流程决策 |
| M5 | 证据继承式实验设计 | 基于 GapCard 设计实验，不重新发明数据集和指标 |
| M5.5 | ResearchGate | 决定通过、回退、补证据、重设实验或终止 |
| M6 | Verification Layer | 不只生成代码，而是生成不同领域的可验证产物 |
| M7 | 论文/报告生成层 | 基于证据链、批判卡和验证结果生成草稿 |
| M8 | Git-like 研究管理层 | fork / replay / compare / merge |
| Appendix A | 自我进化引擎 | 经验记忆、Prompt A/B、SkillLibrary |
| Appendix B | 对抗式数据工厂 | 离线生成红蓝对抗数据和 DPO 偏好数据 |

### 2.2 推荐主流程

```text
启动
→ Direction Intake：询问用户是否已有研究方向
→ 如果没有方向：M0 生成候选 TopicCard / GapCard
→ 如果已有方向：跳过 M0，进入 M1
→ M1 问题精炼
→ M2 多源检索
→ M2.5 文献访问状态解析
→ M3 GapCard 生成
→ M4 Evidence-grounded Roundtable
→ M5 实验方案设计
→ M5.5 ResearchGate
   ├─ pass → M6 Verification Layer
   ├─ revise_experiment → 回 M5
   ├─ fetch_more_evidence → 回 M2 / M2.5
   ├─ refine_question → 回 M1
   ├─ choose_new_topic → 回 M0
   └─ stop → END
→ M7 论文草稿生成
→ Appendix A reflect
```

### 2.3 有 M0 与无 M0 两种运行模式

| 模式 | 候选路线来源 | M8 默认分支起点 | 适用场景 |
|---|---|---|---|
| 有 M0 | M0 自动发现 TopicCard / GapCard | M1 | 用户没有明确课题，希望系统主动选题 |
| 无 M0 | M1 生成多个 PICO / 问题变体 | M1 后或 M2 | 用户已有方向，但希望多路线探索 |

## 3. 模型路由与中转站方案

后续新增架构统一采用 GPT / Claude 中转站，不再把 DeepSeek 作为目标方案默认路径。

### 3.1 核心原则

```text
全部模型调用通过中转站
常规任务走 GPT
关键裁决走 Claude
业务模块只依赖 chat / reasoner / critical 语义角色
模型替换、降级、成本统计集中在 LLM factory / RelayClient
```

### 3.2 推荐角色映射

```text
MODEL_CHAT      = gpt-5.5
MODEL_REASONER  = gpt-5.5
MODEL_CRITICAL  = claude-opus-4.7
```

> 具体模型名以中转站实际暴露名称为准。业务模块不写死模型名，只写 `get_llm("chat")`、`get_llm("reasoner")`、`get_llm("critical")`。

### 3.3 模块分工

| 模块 | 推荐角色 | 推荐模型路径 |
|---|---|---|
| M0 | reasoner / critical | GPT 初筛；Claude 做最终 Top-K 复核 |
| M1 | reasoner | GPT via 中转站 |
| M2 | chat | GPT via 中转站 |
| M3 | chat / reasoner | GPT via 中转站 |
| M4 普通 Reviewer | reasoner | GPT via 中转站 |
| M4 Meta Reviewer | critical | Claude via 中转站 |
| M5 | reasoner | GPT via 中转站 |
| M6 | chat / reasoner | GPT via 中转站 |
| M7 | chat | GPT via 中转站 |
| M8 compare / merge | reasoner / critical | GPT 初评；Claude 复核 |

### 3.4 成本控制策略

即使全部走 GPT / Claude 中转站，也不应所有节点都调用最贵模型。

```text
80%-90% 调用：MODEL_CHAT
10%-20% 调用：MODEL_REASONER
1%-5% 调用：MODEL_CRITICAL
```

critical 节点采用双阶段：

```text
Stage 1：GPT reasoner 初裁
Stage 2：满足触发条件时 Claude critical 复核
```

触发条件：

```text
M4 reviewer 方差过大
final_rating 位于灰区，例如 5.5-7.0
存在 blocking issue 但 Meta 仍建议通过
M8 多分支评分接近，winner 不明显
医学/安全/高成本实验等高风险方向
用户显式要求 strong_review=true
```

## 4. M0：候选课题发现器

### 4.1 定位

M0 不应承诺“自动发现最终课题”，而应定位为：

```text
候选课题发现器 / candidate topic generator
```

它输出多个候选方向，供用户、M4 或 M8 进一步筛选。

### 4.2 输入

```text
用户兴趣领域
目标学科范围
约束条件：时间、成本、数据可用性、是否需要代码验证
已有关键词或论文种子，可选
```

### 4.3 输出：TopicCard

```python
class TopicCard(TypedDict, total=False):
    topic_id: str
    title: str
    research_direction: str
    candidate_question: str
    suspected_gap: str
    key_evidence: list[str]
    novelty_rationale: str
    feasibility_rationale: str
    risk_factors: list[str]
    score: float
```

### 4.4 发现方向的方法

| 方法 | 说明 |
|---|---|
| 研究趋势识别 | 根据近两年论文增长、方法节点增长、benchmark 增长识别热点 |
| 研究空白识别 | 识别“问题被反复提到，但解决方案不足”的区域 |
| GraphRAG 增强检索 | 利用知识图谱路径、邻居和子图发现潜在 gap |
| Web Search 工具层 | 补充最新论文、博客、代码仓库、benchmark 页面 |

### 4.5 与其他模块关系

```text
M0 → M1：把 candidate_question 精炼为正式研究问题
M0 → M2：为候选方向提供初始 query 和 seed papers
M0 → M3：提供待验证的 suspected_gap
M0 → M4：候选方向可先经过快速批判筛选
M0 → M8：Top-K TopicCard 可直接触发多分支探索
```

## 5. M2 / M2.5：证据检索与访问状态层

### 5.1 M2 增强目标

M2 不只是“检索论文列表”，还要为 M3/M4/M5 提供可判断的证据基础：

```text
论文元数据
摘要
引用关系
全文可访问状态
代码 / 数据集 / benchmark 线索
证据等级
```

### 5.2 M2.5 文献访问状态

建议新增：

```python
class EvidenceAccessStatus(TypedDict, total=False):
    paper_id: str
    access_status: str  # fulltext / abstract_only / restricted / failed
    has_code: bool
    has_dataset: bool
    has_benchmark: bool
    evidence_level: str # high / medium / low
    notes: list[str]
```

### 5.3 证据降权规则

```text
fulltext + code + dataset → high
abstract_only → medium / low
restricted 且无可替代证据 → low
关键 baseline restricted → M4/M5 必须提示风险
```

## 6. M3：GapCard 生成层

### 6.1 新定位

M3 从“简单三元组 + research_gaps 字符串”升级为：

```text
Evidence-grounded GapCard generator
```

### 6.2 GapCard 数据结构

```python
class GapCard(TypedDict, total=False):
    gap_id: str
    title: str
    problem: str
    evidence_papers: list[str]
    existing_methods: list[str]
    missing_piece: str
    datasets: list[str]
    baselines: list[str]
    metrics: list[str]
    novelty_score: float
    feasibility_score: float
    evidence_level: str
    risks: list[str]
```

### 6.3 判断逻辑

更稳的 gap 判断不应只靠“图中孤立节点”，而应综合：

```text
问题节点出现频率
解决方法数量
baseline 是否稳定
benchmark 是否统一
近两年是否仍在反复出现
证据是否只有 abstract
是否存在公开数据集和代码
```

## 7. M4：Evidence-grounded Roundtable

### 7.1 新定位

M4 应升级为：

```text
动态评审团 + 证据审查层 + 流程决策器
```

它不直接执行分叉，但要输出可供 M8 消费的决策。

### 7.2 Agent 结构

```text
M4 Evidence-grounded Roundtable
├─ Orchestrator Agent
│  ├─ 读取 refined_question / GapCard / evidence summary / method summary
│  ├─ 判断本次研究最需要哪些评审视角
│  └─ 动态选择 3-5 个 Reviewer Subagents
│
├─ Reviewer Subagents
│  ├─ novelty_reviewer
│  ├─ methodology_reviewer
│  ├─ statistics_reviewer
│  ├─ reproducibility_reviewer
│  ├─ devil_reviewer
│  └─ domain_specific_reviewer
│
├─ Debate / Recheck Layer
│  ├─ 评分分歧大时触发二轮讨论
│  ├─ blocking issue 出现时触发复审
│  └─ 证据不足时要求补证据
│
└─ Meta Reviewer Agent
   ├─ 汇总 CritiqueCard
   ├─ 结合 reviewer_stats / evidence_level
   ├─ 输出 MetaDecision
   └─ 生成 DecisionCard
```

### 7.3 Orchestrator 动态选 Reviewer

当前 Phase 2 已完成：

```text
Orchestrator 动态选择 Reviewer
→ 被选中的 Reviewer 并行输出 CritiqueCard
→ Meta Reviewer 汇总裁决
```

后续升级为基于证据结构选人：

```text
如果 evidence_level 低 → 加 reproducibility / methodology
如果涉及统计显著性 → 加 statistics
如果涉及医学/临床 → 加 clinical_safety_reviewer
如果涉及 RAG/Agent → 加 retrieval_reviewer / agent_system_reviewer / evaluation_reviewer
如果评分分歧大 → 触发 Debate / Recheck
```

### 7.4 M4 输出：DecisionCard

```python
class DecisionCard(TypedDict, total=False):
    passed: bool
    decision: str              # pass / minor_revision / major_revision / reject / stop
    final_rating: float
    recommended_action: str    # continue / refine_question / fetch_more_evidence / rebuild_gap / revise_experiment / stop
    target_node: str           # m0 / m1 / m2 / m3 / m5 / m6 / m7 / end
    branch_count: int
    branch_variants: list[str]
    blocking_issues: list[str]
    required_fixes: list[str]
    reason: str
```

### 7.5 与 M8 的边界

```text
M4 负责判断“哪里不行、下一步去哪”
M8 负责执行“回到哪个 checkpoint、开几条分支、怎么继续跑”
```

## 8. M5 / M5.5 / M6：实验与验证层

### 8.1 M5：继承证据设计实验

M5 不应凭空生成实验元素，而应继承 GapCard 中的：

```text
datasets
baselines
metrics
existing_methods
risks
```

推荐关系：

```text
M5 = GapCard → ExperimentPlan
```

### 8.2 M5.5 ResearchGate

M5.5 是实验方案后的质量门禁，负责判断：

```text
实验是否可执行
证据是否足够
baseline 是否明确
数据集是否可获得
是否需要回退
```

输出动作：

```text
continue_to_m6
revise_experiment
fetch_more_evidence
refine_question
choose_new_topic
stop
```

### 8.3 M6：Verification Layer

M6 不应只理解为“代码生成模块”，而应抽象为：

```text
可验证产物生成层 / Verification Layer
```

不同领域的验证产物不同：

| 领域 | M6 产物 |
|---|---|
| 计算机科学 / AI | 代码原型、toy run、指标计算脚本、benchmark wrapper |
| 医疗 AI / 医学信息学 | 统计分析计划、实验 protocol、样本量估计、数据字段检查脚本 |
| 生物信息学 / 计算生物学 | 数据处理 pipeline、公开数据 notebook、候选基因筛选流程 |

原有 M6 代码生成与沙箱执行应保留，作为 AI / CS 方向的默认 verification backend。

## 9. M8：研究过程回放与分叉

### 9.1 定位

M8 是 Git-like 研究管理层：

```text
fork → replay → compare → merge
```

它不负责判断科学质量，只负责管理研究路线。

### 9.2 有 M0 时的分支方案

```text
用户输入领域或粗粒度方向
→ M0 生成 Top-K TopicCard / GapCard
→ M8 为每个候选方向创建 fork
→ 每条 fork 从 M1 开始跑 M1-M7
→ 每条 fork 内部可根据 M4/M5/M6 决策回退
→ M8 compare
→ merge winner 为主线
```

### 9.3 无 M0 时的分支方案

```text
用户输入 raw_question
→ M1 生成多个 PICO / 研究问题变体
→ M8 为 Top-K 变体创建 fork
→ 每条 fork 独立运行 M2-M7
→ M4/M7 对每条路径评分
→ M8 compare
→ merge winner 为主线
```

### 9.4 分支内部回退

| 失败原因 | 回退节点 | 说明 |
|---|---|---|
| 创新性不足 | M0 | 换候选课题 |
| 问题定义不清 | M1 | 重写 PICO |
| 证据不足 | M2 / M2.5 | 补检索和全文 |
| gap 不成立 | M3 | 重构 GapCard |
| 实验不可执行 | M5 | 重设实验方案 |
| 代码验证失败 | M6 或 M5 | 换实现策略或降级实验 |
| 论文表达不足 | M7 | 重写草稿或补证据 |

### 9.5 merge 第一阶段定义

第一阶段不要做复杂自动融合：

```text
merge = 选择评分最高 / 用户确认的 winner fork，并标记为 mainline
```

后续再升级为：

```text
merge = winner fork 主体 + 其他 fork 的补充证据 / reviewer 建议 / 实验设计优点
```

## 10. Appendix A / B 在新架构中的位置

### 10.1 Appendix A：Self-Evolving Engine

附录 A 已接入主 DAG：

```text
appendix_recall → M1-M7 → appendix_reflect
```

它包括三层：

| 层级 | 能力 | 用途 |
|---|---|---|
| L1 EvolvingMemory | 经验记忆 | 开头召回历史经验，结尾反思沉淀 |
| L2 Prompt A/B | Prompt 变体评分 | 优先使用历史表现更好的 prompt |
| L3 SkillLibrary | 代码技能库 | 沉淀 M6 验证过的可复用 Python 函数 |

在新架构中应增加质量门槛：

```text
只有 M4 final_rating 达标
或 M6 验证成功
或用户确认有价值
才沉淀到长期记忆 / Prompt A/B / SkillLibrary
```

### 10.2 Appendix B：Adversarial Data Factory

附录 B 当前是离线 CLI 工具，不默认进入主 DAG。

用途：

```text
生成红蓝对抗数据
生成 chosen / rejected 偏好数据
评估 M4 / M5 / M0
优化 prompt
沉淀失败案例和质量规则
```

后续可选接入点：

```text
M4 审核不通过 → 触发 Red/Blue 修复
M5 实验方案后 → 对实验方案做 adversarial attack
离线批量生成 M4/M5 eval 数据集
```

## 12. 失败兜底与生产级保障

### 12.1 Graceful Degradation

各模块应保留局部兜底：

| 模块 | 失败场景 | 降级策略 |
|---|---|---|
| M2 | 单源限流 | 使用其他源 + 缓存 |
| M2 | 全源失败 | 返回检索失败卡片，允许用户手动粘贴论文 |
| M3 | 三元组抽取失败 | 跳过该篇；失败率高时降级关键词云 |
| M4 | Agent 超时 | 单 Reviewer 兜底，限制 max_turns |
| M4 | 分歧无法收敛 | 输出多方观点并列，不强求共识 |
| M5 | 自检失败 | 输出警告标签，要求用户确认 |
| M6 | 沙箱失败 | 降级 generate_only，附错误日志 |
| M7 | 章节失败 | 用 TODO 占位，其他章节继续 |
| M8 | checkpoint 失败 | 降级内存态，提醒本次不可回放 |
| 全局 | 模型 API 失败 | 中转站 fallback 或暂停等待恢复 |

### 12.2 Phase 3 基础设施

| 能力 | 作用 |
|---|---|
| LangSmith | Trace、调试、观察每个节点输入输出和 token 使用 |
| Extended Thinking | 关键裁决节点提升推理质量 |
| Budget Guard | 防止 Agent 长跑、死循环或成本失控 |

## 13. 分阶段实施路线

### Phase A：整理模型入口与数据结构

```text
1. 统一 GPT / Claude 中转站 RelayClient
2. 保留 chat / reasoner / critical 语义角色
3. 定义 TopicCard / GapCard / DecisionCard / EvidenceAccessStatus
4. 更新 ResearchState 字段
```

### Phase B：轻量 M0 + GapCard

```text
1. M0 先基于 M2/M3 生成候选 TopicCard
2. M3 从 research_gaps 升级为 GapCard
3. M4/M5 优先读取 GapCard
4. 加入用户选择候选课题节点
```

### Phase C：M4 / M5 / M5.5 质量门禁

```text
1. M4 输入加入 GapCard / evidence summary / access_status
2. M4 输出 DecisionCard
3. M5 继承 GapCard 中的数据集、baseline、metrics
4. 增加 M5.5 ResearchGate
```

### Phase D：M8 多分支探索

```text
1. 有 M0：Top-K TopicCard → 批量 create_fork
2. 无 M0：M1 Top-K PICO → 批量 create_fork
3. 每条 fork 独立跑 M1-M7 或 M2-M7
4. M8 compare 展示评分、证据、实验方案、论文草稿
5. 轻量 merge：标记 winner fork 为 mainline
```

### Phase E：前端与展示

```text
1. 展示 TopicCard / GapCard / Reviewer Cards
2. 展示文献访问状态和证据链
3. 展示 M8 fork tree
4. 增加关键节点用户确认
5. 输出架构演进博客和在线 Demo
```

## 14. 最终定位总结

新架构不是简单“多加一个 M0”，而是一次系统定位升级：

```text
旧系统：用户给定问题 → 单线研究 → 输出方案/论文
新系统：系统发现候选方向 → 多分支探索 → 证据审查 → 回退修正 → 选择最优路线
```

核心闭环：

```text
M0 / M1 产生候选路线
→ M8 创建研究分支
→ M2/M3 构建证据和 GapCard
→ M4 证据驱动审稿并输出 DecisionCard
→ M5/M6 生成可验证实验产物
→ M8 compare / merge
→ Appendix A 沉淀经验
→ Appendix B 离线生成评估与训练数据
```

最终目标：

```text
让 AI Co-Scientist 从“被动科研助理”升级为“主动、多路线、可回放、可进化的科研工作流系统”。
```
