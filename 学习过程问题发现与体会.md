## 3. 什么是 PICO 框架?

### 原始疑问
PICO 是什么?

### 答案
**循证医学 1995 年 Sackett 提出,Cochrane 综述的标准化问题结构**。

4 个要素把模糊问题拆成**可检索**的结构化查询:
- **P**opulation:研究对象
- **I**ntervention:干预
- **C**omparison:对照
- **O**utcome:结局

扩展:PICOT(加 Time)、PICOS(加 Study design)、PECO(Exposure 代替 Intervention)。

### 在我项目的位置
- **定义**:`backend/co_scientist/state/research_state.py` 第 37-54 行 `class PICO(TypedDict)`
- **实现**:`modules/m1_refiner/refiner.py::build_pico()`
- **消费**:m2 拼检索词、m4 Orchestrator 判"是实证/纯理论"、m5 锚定实验设计

### 可带走的结论
- m1 不是"润色问题",是**问题 → 可操作结构体**的翻译
- PICO 是整个 pipeline 的"问题锚点",**Orchestrator 也用它判断需要哪类 Reviewer**

## 5. AutoGen 到底怎么运作的?我需要它吗?

### 原始疑问
是不是 AutoGen 一个 Agent 干完整件事才交给下一个?

### 答案
**AutoGen GroupChat 是轮流发言,不是 A 干完交 B**:

```
Round 1: Agent_A 说话(可能内部 ReAct 循环调工具)
Round 2: Manager 选下一个 → Agent_B
...
直到终止(max_round / TERMINATE / ...)
```

三个关键点:
- 每轮一个 Agent,**工具循环在那一轮内部完成**
- 所有 Agent **共享 chat history** → 有 anchoring bias 风险
- 发言人选择:round_robin / auto(LLM 选) / manual / random

### 我需不需要?不需要。现在更加不需要 — 因为 Phase 2 已经落地了更好的替代方案

1. **我的 m4 要并行+独立+反从众**,AutoGen 共享 history 是反模式
2. LangGraph 吃完外层编排,再上 AutoGen 变三层编排栈
3. `PromptABTester` 动态替换 prompt,AutoGen stateful Agent 不好插拔
4. **Phase 2 的 Orchestrator-Subagent 是比 GroupChat 更现代的范式**(对应 Anthropic 2025.4 博客),已经是"我选择不用 AutoGen"的最佳工程证据

### 可带走的结论
- "会用 AutoGen" 不一定比 "懂得不用 AutoGen" 更值钱
- **刻意选择 vs 堆框架**,选型能力比框架熟练度更能区分档次
- 面试可直接讲:"AutoGen 在我这里是反模式,我选了 Orchestrator-Subagent 并且写了代码"(`m4_critique/orchestrator.py`)

## 6. 什么是 LangGraph DAG?

### 原始疑问
LangGraph DAG 是什么?

### 答案
**LangGraph DAG = LangChain 官方多 Agent 编排框架 + 有向无环图**

三个核心概念:
- **节点(Node)**:普通 Python 函数,读写 state
- **边(Edge)**:`add_edge` 固定 / `add_conditional_edges` 按 state 路由
- **状态(State)**:TypedDict,所有节点共享读写

**相比"写一个大函数"的优势**:

| 问题 | 图编排怎么解 |
|---|---|
| 中途挂了要从头 | Checkpointer 断点恢复(我项目三级降级:Postgres→SQLite→Memory)|
| 暂停等用户决策 | `interrupt_before=["m6_execute"]` 原生支持 |
| 动态分支 | `conditional_edges` |
| 回放历史某一步 | checkpoint 存了每步完整 state |
| 流式输出 | `graph.astream()` |
| 成本失控 | Phase 3 加的 `budget_guard(RUN_BUDGET_USD)` 包裹整条 invoke |
| Trace 可观测 | Phase 3 加的 LangSmith,`build_graph()` 入口自动启用 |

### 在我项目的位置
- `backend/co_scientist/graph.py` — 10 节点 DAG 的教科书实现
- `教学资料/03-Agent框架/README.md` — 详细讲解

### 可带走的结论
- DAG 把**业务逻辑**和**运行时保障**拆开
- 节点函数只管输入/输出,框架管持久化/恢复/中断/流式/分支
- 我已经把"成本护栏 + 观测性"也加到 DAG 层

## 9. 什么是 RRF 融合排序?

### 原始疑问
RRF 是什么?

### 答案

**RRF = Reciprocal Rank Fusion,倒数排名融合**。

公式:`score(d) = Σ 1/(k + rank_i(d))`,k=60(TREC 原论文经验值)。

**核心思想**:**只用 rank 不用原始分数**,避免不同源分数不可比。

### 最小例子
```
路A(BM25):1.P1  2.P2  3.P3
路B(向量):1.P3  2.P1  3.P4

k=60:
P1 = 1/61 + 1/62 ≈ 0.0325  (两路都上榜,且靠前)
P3 = 1/63 + 1/61 ≈ 0.0323
P2 = 1/62       ≈ 0.0161  (只一路)
P4 = 1/63       ≈ 0.0159

→ P1 > P3 > P2 > P4
```

### 为什么赢
- 不同源量纲完全不可比(BM25 vs cosine vs relevance score)
- 归一化到 [0,1] 会放大噪音
- "只用 rank = 承认我不知道绝对相关度,但相信各源内部排序"

### 在我项目的位置
- `backend/co_scientist/modules/m2_retriever/fusion.py::reciprocal_rank_fusion()`
- 流程:三源(arXiv / S2 / OpenAlex)**可选走 MCP Server 模式**(Phase 1 新增)→ 并行搜 → RRF 融合 → 时间衰减 → top_k

### 可带走的结论
- 工业界(ES / Vespa / Weaviate / LangChain EnsembleRetriever)默认用 RRF,简单 + 稳
- 面试讲法:"三源分数分布不同,不可比。RRF 抛掉绝对分只保留 rank,工程上最稳。k=60 沿用 TREC 经验没做 ablation 所以没改"

## 11. 2026 年最新的 Agent 架构是什么?

### 原始疑问
你说的 Compound AI 是 2024 的,2026 年最新啥样?

### 答案

**2025-2026 的六个主要演进**:

### (1) MCP(Model Context Protocol)成为事实标准协议 ⭐最重要
- 2024.11 Anthropic 发布
- 2025 被 OpenAI / Google / Cursor / Zed 全部跟进
- 工具只写一次 MCP Server → 任何 MCP 客户端都能用
- **已经是业界事实标准**(类比 LSP 之于编辑器)

### (2) Orchestrator-Subagent 成为多 Agent 主流形态
- Anthropic 2025.4《How we built our multi-agent research system》
- Claude Code 的 subagent 架构、OpenAI Deep Research、Perplexity Pro

### (3) Context Engineering 取代 Prompt Engineering

## 12. 工具放外面,Agent 就不算"工具调用"了?这是退化吗?

### 原始疑问
工具都用代码实现了,Agent 没做工具调用,算退化吗?

### 答案

**这是进化,不是退化**。业界术语叫 **Orchestrator-Worker 模式 / Tool-Isolated Architecture**。

### 两种架构对比

| 维度 | Tool-Embedded(ReAct / AutoGen) | **Tool-Isolated(我的架构)** |
|---|---|---|
| 每个 Agent | 带工具,自主 ReAct 循环 | 只做判断,工具由编排层调度 |
| 工具调用次数 | Agent 自主决定,不可预测 | **DAG 写死,可预算** |
| 多 Agent 输入 | 各自搜不同内容 | **统一输入,公平** |
| 观测性 | Agent 内部黑盒 | 节点级清晰,LangSmith 全链路 |
| 成本 | 可能失控 | **Budget Guard 硬上限** |

### 我项目的工具层定位

Phase 1 之前:工具直接在代码里(`sources/arxiv_src.py` 等)
Phase 1 之后:**工具抽象成 MCP Server**,主程序通过 `mcp_client.py` 调用,feature flag 切换。

这是工具层的**"协议化"升级** —— 不是"把工具交给 Agent",而是"把工具标准化让更多 Agent 能用"。

### 成熟度三级

- **L1 Prompted Agent**:只调 LLM
- **L2 Orchestrated Agent**(我的):工具在编排层/MCP,Agent 只判断 ⭐
- **L3 Autonomous Agent**:带工具 + ReAct 循环 + 自主规划

L3 炫酷但**不可控**,Anthropic 2024《Building Effective Agents》明确说:**"很多情况下,workflow 优于 agent"**。

### 可带走的结论
- L2 是**生产甜蜜点**,不是能力不足
- 我的 Reproducibility Reviewer 就是**预留的 L3 扩展位**(未来可以让它真 clone repo 跑 readme)
- 面试话术:"我刻意选择,不是能力不足。Phase 1 的 MCP 让工具层独立标准化,这比给 Reviewer 塞工具更工程化"

## 13. 从手搓 → 框架 → 编排,是不是绕回来了?

### 原始疑问
v1 手搓 → v2 框架 → v3 又写很多代码,是不是绕了一圈?

### 答案

**不是回退,是代码的"层次"不一样了**。

| 版本 | 代码在做什么 | LLM 在做什么 |
|---|---|---|
| v1 手搓(agent1) | **业务规则**(出题/评分/流程)| 基本没有 |
| v2 框架(agent1_upgrade) | 几乎不写 | **承担一切** |
| v3 编排(agent3 当前) | **基础设施**(RRF / DAG / MCP / Orchestrator / Budget Guard / Observability) | **智能判断** |

**v1 代码 ≠ v3 代码**:
- v1 代码 = 业务规则(随项目死)
- v3 代码 = **工程能力**(可复用一辈子)

### 对应业界演进线
Berkeley BAIR 2024《The Shift from Models to Compound AI Systems》:
```
Stage 1: Rule-Based Systems   ← v1
Stage 2: End-to-End LLM Agent ← v2
Stage 3: Compound AI Systems  ← v3(我现在,业界主流)
```

### 软件史上类似的"轮回"
- 高级语言 → 又写 C/Rust?**是学会"何时用底层"**
- ORM → 手写 SQL?**是学会"何时用高层"**
- LLM 做一切 → 代码 + LLM 分工?**是学会"何时用代码"**

### 工程师级别对照
- v1 想法:"LLM 不够用,自己写" → 初级
- v2 想法:"这个也交给 Agent?" → 中级
- **v3 想法:"代码做更稳,那件事交给 LLM" → 高级(拿捏分寸)**

### 可带走的结论
- **走过 v2 才能真懂 v3 的价值**
- 简历叙事可以写:"踩过 v2 坑,理解为什么 v3 更好"—— 比"我用 LangGraph"值钱 10 倍
- Phase 1-3 的三次升级**每一次都在强化 v3 叙事**(MCP 是工具层标准化、Orchestrator 是协作范式升级、Budget Guard 是运行时保障)

## 15. Orchestrator-Subagent 和 AutoGen GroupChat 有什么本质区别?

### 原始疑问
这俩都是"多 Agent"?

### 答案

| 维度 | AutoGen GroupChat(2023) | **Orchestrator-Subagent(2025,我的)** |
|---|---|---|
| 谁决定下一步 | Manager 每轮选发言人 | Orchestrator **开场一次性定 team** |
| 上下文共享 | 全员共享 history | **每个子 Agent 独立上下文** |
| 主 Agent 看到 | 全员的对话流 | **只有每个子 Agent 的最终结论** |
| 发言 | 轮流 | **并行**(asyncio.gather) |
| 适合 | 开放式讨论/辩论 | **结构化并行任务(评审/研究)** |
| 代表 | AutoGen、CAMEL | Anthropic 2025 博客、Claude Code subagent、Perplexity Pro |

### 我的实现(`m4_critique/orchestrator.py`)

```python
def select_reviewers(refined_question: str, method_summary: str) -> dict:
    """Orchestrator:LLM 看任务性质,动态选 3-5 个 Reviewer。"""
    llm = get_llm("chat")  # 轻量决策,成本 ~$0.0005
    try:
        result = llm.chat_json(...)
    except Exception:
        return _fallback_all_reviewers()  # 降级全量

    cleaned = _sanitize_selection(result.get("reviewers", []))
    # 清洗规则:过滤非法 / 去重 / devil 强制 / 3-5 数量边界
    return {"reviewers": cleaned, "reason": result.get("reason"), "fallback": False}
```

### 设计亮点

1. **devil 硬编码必选**(`_MANDATORY_NAME = "devil"`):防止全员温和共识泡沫
2. **3-5 数量边界**:少于 3 不多视角,多于 5 等于没选
3. **失败降级全量**:宁可贵不要卡住
4. `settings.M4_USE_ORCHESTRATOR` feature flag 默认 True,可关闭对比旧行为

### 测试覆盖(`test_orchestrator.py` 12 测试)

- Part 1 清洗(6):非法名/大小写/去重/devil 强制/MIN 补齐/MAX 截断
- Part 2 集成(4):正常选择/LLM 异常降级/bad schema 降级/resolve 容错
- Part 3 roundtable(2):Orchestrator 选择生效/settings 开关真跳过

### 可带走的结论
- AutoGen GroupChat 是"群聊",适合自由辩论
- Orchestrator-Subagent 是"派遣",适合并行独立任务
- **我的评审场景要独立意见,不是辩论** → 选后者才对
- 这个升级**有代码、有测试、有论文对应**,面试是硬通货

### 答案(基于 Phase 3 落地代码)

### Extended Thinking(Claude 推理预算)
**解决**:复杂推理任务模型"想得不够深"就给答案。

**我的实现**(`llm/claude.py::chat`):
```python
def chat(self, messages, *, purpose="", thinking_budget=None, ...):
    # 按 purpose 智能选择:meta 节点自动开,其他默认关
    if budget is None:
        if "meta" in (purpose or "").lower():
            budget = settings.CLAUDE_THINKING_BUDGET_META  # 建议 4000
        else:
            budget = settings.CLAUDE_THINKING_BUDGET_DEFAULT  # 0

```

**分级**:`purpose` 含 "meta" 自动启用,普通调用关闭。
**硬约束**:`temperature=1.0`,`budget < max_tokens`。

### Budget Guard(成本护栏)
**解决**:Agent 长跑 bug → LLM 循环反复调 → API 账户被打爆。

**我的实现**(`utils/budget_guard.py`):
```python
import contextvars
_RUN_SPENT = ContextVar("run_spent", default=0.0)
_RUN_BUDGET = ContextVar("run_budget", default=0.0)

@contextmanager
def budget_guard(limit_usd: float):
    # 进入时重置累计,退出时恢复
    ...

def charge(cost_usd: float) -> None:
    budget = _RUN_BUDGET.get()
    if budget <= 0:
        return
    new_total = _RUN_SPENT.get() + cost_usd
    _RUN_SPENT.set(new_total)
    if new_total > budget:
        raise BudgetExceeded(spent=new_total, budget=budget)
```

**挂载点**:`utils/cost_tracker.py::add()` 在每次 LLM 调用落库后调 `charge(cost)`。
**入口包裹**:`graph.py::run_pipeline()` 用 `with budget_guard(settings.RUN_BUDGET_USD)` 包住整条 `graph.invoke`。
**为什么 ContextVar**:asyncio 并发安全,FastAPI 多请求互不偷钱。
**对标**:Devin / Cognition。

### LangSmith(观测性)
**解决**:昨天挂的 run 是哪一步挂的?改 Prompt 后效果好还是坏?

**我的实现**(`utils/observability.py`):
```python
def setup_langsmith() -> bool:
    if not settings.LANGSMITH_TRACING:
        return False
    if not api_key:
        return False  # 没 Key 静默跳过
    # 同时 export LANGCHAIN_* 和 LANGSMITH_*(兼容期)
    os.environ["LANGCHAIN_TRACING_V2"] = "true"
    os.environ["LANGSMITH_API_KEY"] = api_key
    ...
```

**调用点**:`graph.py::build_graph()` 入口自动 `_try_setup_langsmith_once()`。
**特性**:idempotent(多次调用安全)、无 Key 静默跳过、延迟初始化避免 import side effect。

### 三件事的关系

```
run_pipeline(q, budget=1.0)
  ├─ setup_langsmith()          ← 观测:trace 自动上报
  └─ with budget_guard(1.0):     ← 成本:run 级硬上限
       graph.invoke(...)
         └─ m4 Meta Claude
              └─ thinking_budget=4000   ← 推理:按 purpose 分级
```

**三件事正交**:观测只看 / 推理只改单次调用 / 护栏只中断,互不影响。

### 测试覆盖
`test_phase3_infra.py` 14 测试(4+4+6)全过。

### 可带走的结论
- 这三件是"Demo → 生产"的分水岭,**不做不出事,做了面试加分**
- 面试话术:"三层生产级基础设施 —— **能看见(LangSmith)/ 能兜住(Budget Guard)/ 能深思(Extended Thinking)**"
