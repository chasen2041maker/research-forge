## 3. ⭐ 必读 4 个核心文件（优先级排序）

### ① `backend/co_scientist/graph.py` —— 主编排（最先读）

**关键点**：
- `build_graph()` 用 `StateGraph(ResearchState)` 把 8 模块拼图。
- `safe_node()` 装饰器：每个节点包 try/except，失败写 `error_log` 不炸流程。
- `interrupt_before=["m6_execute"]`：代码执行前**暂停**，等用户选执行档位（Human-in-the-loop）。
- `_make_checkpointer()`：三级降级 **Postgres → SQLite → Memory**，是回放/分叉的底层。
- `run_pipeline()`：`thread_id = fork_id`，用 thread_id 区分不同研究分支。

**话术**：
> "用 StateGraph 把 8 节点串成图。三个关键设计：① safe_node 装饰器保证失败不中断；② 代码执行前 interrupt_before 暂停等用户决策；③ Checkpointer 三级降级，生产用 Postgres 支持分叉回放。"

---

### ② `backend/co_scientist/state/research_state.py` —— 全局状态

**关键点**：
- 用 `TypedDict` 定义 `ResearchState`（LangGraph 原生支持，比 dataclass 轻、比 pydantic 快）。
- **核心 Reducer 机制**：`papers: Annotated[list[Paper], operator.add]` —— 并行节点写同一字段时**合并**而非覆盖。
- 子结构：`PICO / Paper / Triple / CritiqueCard / Experiment / CodeArtifact / PaperDraft`。

**话术**：
> "LangGraph 核心思想是节点共享 State。关键是 **Reducer** —— M2 检索会并行调 3 个源都写 papers，不用 `Annotated[list, operator.add]` 就会互相覆盖。"

---

### ③ `backend/co_scientist/modules/m4_critique/` ⭐⭐ 项目最大亮点

**关键点（Phase 2 升级后）**:
- **Orchestrator 动态选 Reviewer**（`orchestrator.py::select_reviewers`）：LLM 看问题性质，从候选 5 类中选 3-5 个，**devil 强制必选**
- **5 个 Reviewer 人设**：novelty / methodology / statistics / reproducibility / devil（在 `reviewers.py::REVIEWER_REGISTRY`）
- `run_reviewers_parallel(personas=...)`：`asyncio.gather` 并行被选中的 Reviewer，上下文严格隔离（反 anchoring bias）
- `compute_variance()`：rating 方差 > 2 触发**二次辩论**，Devil's Advocate 看到 Round 1 摘要再评一轮
- `meta_decide()`：Claude Opus 终裁 + **Extended Thinking 4000 token 推理预算**（Phase 3）；**降级**：Claude 失败 → DeepSeek-reasoner 兜底
- meta_decision 里附着 `orchestrator` 信息（召了谁 + 为什么），下游可审计

**话术（重点打磨，Phase 2+3 升级版）**：
> "这是项目最能体现 Agent 工程深度的模块，也是我 Phase 2 升级的核心。老版本是永远跑全 5 个 Reviewer，新版本是 **Orchestrator-Subagent 范式** —— Orchestrator LLM 先看问题性质，动态决定召哪 3-5 个 Reviewer，对应 Anthropic 2025.4 多 Agent 研究系统博客。**devil 永远必选**，这是硬编码兜底防全员温和共识。然后并行独立评审（严格隔离上下文防 anchoring bias）。方差 >2 触发 Devil's Advocate 看 Round 1 摘要二辩。Meta-Reviewer 用 Claude Opus 终裁并启用 Extended Thinking 给 4000 token 推理预算，Claude 挂了自动降级 DeepSeek-reasoner 保流程不断。整个 m4 在 `budget_guard` 上下文里跑，超限抛 BudgetExceeded。"

---

### ④ `backend/co_scientist/modules/m2_retriever/retriever.py` —— RAG 亮点 + MCP 标准化

**流水线**：
1. **Query Rewriting**：LLM 把 1 个原问题改写成 5 条英文 query
2. **并行多源**：每 query × 3 个源（arXiv / Semantic Scholar / OpenAlex）= 15 并发
3. **RRF 融合**：`score = Σ 1/(k + rank_i)`，k=60
4. **时间衰减**加权（新论文加分）
5. **Citation Chasing**：top-5 扩展引用
6. **再 RRF** 一次，截断到 top_k

**Phase 1 升级（MCP 标准化）**:
- 检索源独立成 **MCP Server**（`mcp_servers/arxiv_server.py` 等 3 个）
- `mcp_client.py` 提供和原函数同签名的 async 接口
- `retriever.py` 顶部按 `settings.USE_MCP` feature flag 分流：
  ```python
  if settings.USE_MCP:
      from co_scientist.modules.m2_retriever.mcp_client import (
          search_arxiv, search_openalex, search_semantic_scholar,
      )
  else:
      from co_scientist.modules.m2_retriever.sources import (...)
  ```
- 下游 `asyncio.gather(...)` 一字不改
- **能直接挂到 Claude Desktop**（`npx @modelcontextprotocol/inspector python -m co_scientist.modules.m2_retriever.mcp_servers.arxiv_server`）

**话术**：
> "完整高级 RAG。用 asyncio.gather 把 15 个并发请求压到 3 秒。`return_exceptions=True` 保证单源挂掉不影响全局。RRF k=60 是 Cormack 2009 经验值。Phase 1 我把 3 个检索源独立成了 MCP Server，对应 Anthropic 2024.11 发布的 Model Context Protocol。feature flag 切换，原代码一字未改，向后兼容。这让我的检索源可以被 Claude Desktop / Cursor 这些工具直接复用——这比锁在项目内部有用得多。"

## 5. 高频追问预案（背 8 条保底）

**Q: LangGraph 和 LangChain 区别？**
A: LangChain 是链（Runnable 串联），LangGraph 是图（有环、条件边、状态）。我这有循环（Critique 不过重写）和分叉（fork），必须用图。

**Q: 为什么不用 AutoGen / CrewAI？**
A: 三点。① **范式错配**：AutoGen GroupChat 是共享上下文+轮流发言，我的评审场景要**刻意隔离上下文**避免 anchoring bias，AutoGen 反而是反模式。② **LangGraph 已经吃完外层编排**，再加 AutoGen 变三层编排栈，调试/checkpoint/成本全割裂。③ **我升到了更现代的 Orchestrator-Subagent**（Phase 2），对应 Anthropic 2025.4 博客——主 Agent 一次性派发子 Agent，并行独立，比 GroupChat 轮流发言更适合评审场景。

**Q: MCP 是什么？为什么要接？** ⭐（Phase 1 关键题）
A: Model Context Protocol，Anthropic 2024.11 发布的 Agent 工具/上下文标准协议，2025 被 OpenAI / Google / Cursor / Zed 全部跟进，已经是**业界事实标准**。类比：LSP 之于编辑器、USB 之于硬件。我把 3 个检索源独立成 MCP Server，效果是任何支持 MCP 的客户端（Claude Desktop、Cursor、其他团队的 Agent）都能直接用我的工具，不必重复造轮子。feature flag 切换，向后兼容。这让我在 2026 招聘坐标里比不接 MCP 的项目高一档。

**Q: Orchestrator-Subagent 和 AutoGen GroupChat 到底区别在哪？** ⭐（Phase 2 关键题）
A: 四个维度全对立。① **谁决定**：AutoGen 是 Manager 每轮选发言人，Orchestrator 是开场一次性定 team。② **上下文**：AutoGen 全员共享 history，Orchestrator **每个子 Agent 独立上下文**。③ **主 Agent 看到**：AutoGen 看完整对话流，Orchestrator 只看每个子 Agent 的最终结论。④ **发言**：AutoGen 轮流，Orchestrator 并行。我的评审场景要**独立意见不是辩论**，选 Orchestrator 才对。

**Q: 怎么防止 Orchestrator 选出"全员温和"的 Reviewer？**
A: 硬编码 `devil 必选`（`_sanitize_selection` 强制注入）。Orchestrator 偶尔会选和事老组合，devil 作为魔鬼代言人必须在场，这是结构性保障。

**Q: Extended Thinking 是什么？怎么用？** ⭐（Phase 3）
A: Claude Sonnet 4+ / Opus 4 的显式推理预算 API（2025 发布），类似 o1/o3 的 reasoning tokens。传 `thinking={"type": "enabled", "budget_tokens": 4000}`，模型先花 4000 token 内部思考再回答。我按 `purpose` 分级：meta 终裁这种高风险决策自动开，普通调用不开。硬约束：temperature 必须 1.0，budget < max_tokens。

**Q: Budget Guard 怎么实现？为什么用 ContextVar？** ⭐（Phase 3）
A: `budget_guard(limit_usd)` 作为 contextmanager 包住整条 run，`cost_tracker.add()` 里每次 LLM 调用后调 `charge(cost)` 累加，超限抛 BudgetExceeded。用 **ContextVar** 而非全局变量因为 asyncio 并发时 FastAPI 多请求互不"偷钱"，每个 run 自己的 context 互相隔离。超限用 `raise` 不用 flag——立刻中断，safe_node 捕获写 error_log，流程优雅结束。对标 Devin/Cognition。

**Q: LangSmith 怎么接的？**
A: env 驱动，`build_graph()` 入口自动调 `setup_langsmith()`，export `LANGSMITH_*` 环境变量后 LangChain SDK 自动上报 trace。特点：idempotent（多次调用安全）、无 Key 静默跳过、延迟初始化防 import side effect。**能在面试时直接分享 trace 链接**。

**Q: 记忆系统越用越大怎么办？** ⭐
A: 我做了**基于时间 + 使用频次的遗忘机制**。`recall` 命中自动 `used_count += 1`；`forget_stale(max_age_days=90, min_uses=1)` 删除"又老又没被召回"的双重噪音记忆。用 AND 不是 OR 避免误杀"老但经典"的经验。对应 MemGPT 分页驱逐思想的极简版。

**Q: 怎么保证 Agent 输出质量不回归？** ⭐
A: 我有 **Agent Evals 套件**三层覆盖。Schema Eval 硬校验字段；Consistency Eval 跑 3 次看 Meta rating 标准差；Quality Eval 用 Claude Opus 做 LLM-as-Judge 按 rubric 打 1-5 分。改 Prompt 都会跑 eval，标准差飙升要回滚。Mock 模式零成本可跑 CI，真实模式 ~$0.25 一次。**把 Prompt 工程从凭感觉变成有量化指标**。

**Q: 多 Agent 怎么避免互相吹捧？**
A: 三招：① Reviewer 人设明确要求找问题；② Devil's Advocate 专门唱反调；③ Meta-Reviewer prompt 里强调"分歧点优先"。

**Q: 为什么双模型（DeepSeek + Claude）？**
A: 成本质量权衡。95% 调用走 DeepSeek（$0.14/M token）做生成推理；5% 关键决策（Meta-Reviewer、论文润色）用 Claude Opus 兜底，单次 <$0.6。

**Q: Reducer 是什么？**
A: LangGraph 里 `Annotated[list, operator.add]` 告诉框架：多个并行节点返回该字段时**合并**而非覆盖。M2 并行调 3 个检索源必须用它。

**Q: Checkpoint 怎么实现分叉？**
A: PostgresSaver 每节点自动 snapshot state。`graph.invoke(config={"thread_id": "new_fork", "checkpoint_id": "xxx"})` 从历史点创建新线程。

**Q: RRF 的 k 为什么选 60？**
A: Cormack 2009 论文经验值，对排名靠后的结果惩罚更平滑。没做大规模调参。

**Q: Neo4j 知识图谱怎么构建？**
A: LLM 抽取 `(head, relation, tail)` 三元组，Cypher `MERGE (a)-[:REL]->(b)` 写入。Schema 半结构化（论文/作者/方法/数据集/指标）。

**Q: 代码沙箱怎么保证安全？**
A: Docker 容器隔离 + `--network none` + CPU/内存限制 + 超时 kill。LLM 代码只能读 mount 进来的数据目录。

**Q: 前端怎么可视化 LangGraph？**
A: 后端 SSE 推送每个节点的 `on_node_start / on_node_end`，前端 React Flow 画有向图，节点状态实时染色。

**Q: 项目最难的点？**
A: Checkpoint 一致性 —— 节点内部多次 LLM 调用时如何保证中间失败能恢复。解决：把每次 LLM 调用结果也纳入 state，而不是局部变量。

**Q: 有什么没做好？**
A: ① 没做评测基准，质量靠人工打分；② 知识图谱 schema 是 ad-hoc，没做本体对齐；③ 代码沙箱不支持 GPU。

### M1 · 问题精炼 (`m1_refiner/refiner.py`)

> "M1 把用户输入的模糊问题转成结构化的研究问题。具体做法是用 **PICO 框架**（临床研究常用，我借来描述 NLP 问题）拆成 Population / Intervention / Comparison / Outcome 四个维度。
>
> 实现是三段式：先用 deepseek-reasoner 判断问题是否够具体，**温度调到 0.2**（判断类任务温度低）；不够具体就生成澄清问题反问用户，CLI 模式直接打印、Web 模式 SSE 推到前端；信息收齐后再用 LLM 生成 PICO 结构化输出。
>
> 这里用 reasoner 而不是 chat 是因为'判断+反问'是推理任务，需要更强的逻辑能力。"

---

### M2 · 文献检索 (`m2_retriever/retriever.py`) ⭐

> "M2 是完整的高级 RAG 流水线，6 步：
> 1. **Query Rewriting**：LLM 把原问题改写成 5 条英文 query；
> 2. **多源并行检索**：每条 query × 3 个学术 API（arXiv / Semantic Scholar / OpenAlex）= 15 个并发请求，用 `asyncio.gather(*tasks, return_exceptions=True)` 串起来，串行 30 秒压到 3 秒，单源挂掉不影响整体；
> 3. **RRF 融合**：`score = Σ 1/(k + rank_i)`，k=60（Cormack 2009 经验值），跨源排名合并；
> 4. **时间衰减**加权（新论文加分）；
> 5. **Citation Chasing**：top-5 论文扩展引用；
> 6. **再 RRF** 一次返回 top-30。
>
> State 里 `papers` 字段用 `Annotated[list, operator.add]` Reducer，并行节点写入自动合并。"

---

### M3 · 知识图谱 (`m3_kg/kg_builder.py`)

> "M3 是 GraphRAG 的基础组件，把检索到的论文变成知识图谱。
>
> **抽取**：每篇论文喂给 LLM，让它输出 `(head, relation, tail)` 三元组，并行调用。**关系白名单** 6 种：improves / uses / compares_with / cites / proposes / evaluates_on，不在白名单的丢弃，避免关系无限发散。
>
> **存储**：默认 NetworkX（轻量、零部署），生产模式切 Neo4j Cypher `MERGE (a)-[:REL]->(b)` 写入。
>
> **附加价值——研究空白识别**：图里有'问题/挑战'类节点但没对应'方法'节点，启发式标为研究空白，喂给后面的 m4 和 m5。"

---

### M4 · 多 Agent 批判辩论 (`m4_critique/roundtable.py`) ⭐⭐ 最大亮点

> "M4 是项目最能体现多 Agent 工程的模块。
>
> **5 个人设 Reviewer 并行评审**：novelty / methodology / statistics / reproducibility / devil（唱反调），用 `asyncio.gather` 并行跑，10 秒压到 2 秒。
>
> **方差驱动二次辩论**：算 5 张评审卡 rating 的方差，>2 说明分歧大，触发 Devil's Advocate 第二轮 —— 把 Round 1 的评审摘要喂给它，让它针对分歧点再发言一次。
>
> **Meta-Reviewer 终裁**：把所有卡片喂给 **Claude Opus 4.7** 出最终决定（rating + decision + 改进建议）。**降级策略**：Claude API 失败回落到 deepseek-reasoner 兜底，保证流程不断。
>
> 用 Claude 而不是 DeepSeek 做终裁，是因为这是关键决策节点，质量优先于成本。"

---

### M5 · 实验方案 (`m5_experiment/designer.py`)

> "M5 把研究问题变成可执行实验方案。
>
> 用 deepseek-reasoner 一次结构化生成，输出 `Experiment` schema：name / datasets / baselines / metrics / expected_results / ablations / statistical_test。
>
> **关键是自检 Agent**：照着 ML reproducibility checklist 检查方案 —— '有数据集 / 至少 2 个基线 / 有评测指标 / 有消融 / 有显著性检验'，5 项必须全过。不过的项让 LLM 补。
>
> 真实生产可以扩成 ReAct 让 Agent 调 HuggingFace Datasets 或 Papers With Code 的 API 查 SOTA，目前是 LLM 直出版。"

---

### M6 · 代码沙箱 (`m6_code/code_gen.py`)

> "M6 把实验方案变成可运行 PyTorch 代码。设计上**两段式**：m6_generate 生成代码，m6_execute 沙箱跑，分两个节点是因为可以在中间 interrupt 等用户决策。
>
> **三档执行开关**：
> - `generate_only`：只生成不跑（默认，最安全）
> - `dry_run`：跑语法检查 + import 测试
> - `full_execute`：Docker 沙箱里跑全套
>
> **沙箱安全四件套**：Docker 容器隔离 + `--network none`（断网防恶意请求）+ CPU/内存限制 + 超时 kill。LLM 生成的代码只能读 mount 进来的数据目录。
>
> graph.py 里 `interrupt_before=['m6_execute']` 在跑代码前**强制暂停**，等用户在前端选档位，这是 Human-in-the-loop。"

---

### M7 · 论文初稿 (`m7_writer/writer.py`)

> "M7 用多 Agent 并行写一篇 IMRaD 结构论文。
>
> 流程：① **Style Guide Agent** 先定调（确定论文写作风格）；② **6 个章节 Agent 并行**写 abstract / introduction / related_work / method / experiments / discussion，每个吃同一份 style guide 保证风格统一；③ **Editor Agent**（Claude Opus）统一润色。
>
> **防幻觉**：引用强制只能引 references 数组里的论文，配套 `citation_verify.py` 校验所有 cite 是否在白名单。
>
> 输出 LaTeX 主文件 + BibTeX，可直接 Overleaf 编译。"

### M8 · 分叉回放 (`m8_replay/fork_manager.py`)

> "M8 把 LangGraph 的 Checkpointer 包装成研究分叉树。
>
> 4 个操作：
> - **replay**：回到任意节点重跑（调试不用从头来，省钱）；
> - **fork**：从某节点拉新分支探索不同假设，比如同一个问题用不同实验设计跑两个分支；
> - **compare**：横向对比多分支成果；
> - **merge**：好的子路径合回主线。
>
> 实现上 `ForkManager` 用独立 SQLite 存分叉元数据（fork_id / parent / branch_node / final_rating / status），真正的 state 快照由 LangGraph 的 PostgresSaver 负责。前端用 React Flow 把分叉关系画成树。"

### 附录 A1 · Reflexion 经验记忆 (`appendix/evolve/memory.py`)

> "附录 A 借鉴 Reflexion 思想：任务结束后让 LLM 反思 → 提炼 5 类经验（domain / strategy / failure / user / tool）→ 存 SQLite → 下次任务自动召回。
>
> 召回走 DeepSeek embedding 做余弦相似度（阈值 0.3），embedding 不可用时自动降级到词袋。
>
> **已挂入主 DAG**：`START → appendix_recall → m1…m7 → appendix_reflect → END`。recall 写入 `state.recalled_memories`，被 m5_experiment 消费；reflect 在主流程末尾统一反思入库。两个节点都走 `safe_node` 包装，LLM/SQLite 出错不会断主流程。"

### 附录 A2 · Prompt A/B 进化 (`appendix/evolve/prompt_ab.py`)

> "思路来自 DSPy / OPRO：跑评测集记录每个 prompt 的表现 → 失败时让 LLM 改 prompt → A/B 测试 → 胜者上位。
>
> **闭环已打通**：
> - m5_experiment 运行时调 `best_for('m5_experiment')` 选当前最优变体，没变体就回退到硬编码 SYSTEM；
> - 主 DAG 的 `appendix_reflect` 节点在流程结束时，按 `meta_decision.final_rating` 自动 `record_score` 回写；
> - `evolve_prompt` 基于失败案例让 reasoner 改 prompt 并自动注册；
> - CLI 命令：`prompt-ab-register / -best / -score / -evaluate / -evolve`，`prompt-ab-evaluate` 能对未评测变体用 Claude Judge 自动打分。
>
> **简化点**：没做多臂老虎机/EXP3 这种高级路由，`best_for` 只按平均分挑第一名；评测集要用户自己提供问题列表。"

### 附录 A3 · L3 工具/技能自生成 (`appendix/evolve/skill_library.py`)

> "L3 借鉴 Voyager 思想：Agent 解决任务的过程中会写很多小函数，通过沙箱验证的就注册到技能库，下次相似任务先查库复用，不用每次让 LLM 从零再写一遍。
>
> **和 m6_code 双向挂接**：
> - 生成前：`generate_code` 先用"实验名 + 研究问题"做 query，`SkillLibrary().retrieve()` 词袋召回 top-3 技能签名，拼进 system prompt —— LLM 看到"已有 rrf_fusion(result_lists, k) 这样的工具"就会直接调而不是重写。
> - 执行后：`full_execute` 沙箱 exit=0 时，`_register_verified_skills` 扫描所有 .py，把"非 main-ish 名字 + 函数体 ≥ 3 行"的顶层函数注册入库。
>
> **为什么 full_execute 才注册，dry_run 不注册**：dry_run 只过语法，无法保证逻辑正确，宁可少收也不能把"看起来对其实错"的函数污染库。
>
> CLI：`skill-list / skill-show / skill-delete` + `evolve-dashboard` 能看全库。
>
> **三层合在一起**：L1 记'经验'、L2 记'表达'、L3 记'工具'，三层记忆各管一类可复用资产。"

### 附录 B · Red/Blue 对抗数据工厂 (`appendix/adversarial/red_blue.py`)

> "附录 B 借鉴 GAN 思想做对抗式数据生成：
> - **Blue Team**（deepseek-chat，温度 0.6）：提研究方案 / 修复漏洞
> - **Red Team**（deepseek-reasoner，温度 0.8）：找 Blue 方案的 3 个致命漏洞和攻击路径
> - **Judge**（Claude Opus，温度 0.3）：评分 + 输出偏好对（chosen 是 blue_original 还是 blue_fixed）
>
> **支持单轮和多轮两种模式**：
> - 单轮（`run_round`）：Pairwise，产 1 条 DPO 对；
> - 多轮（`run_multi_round`）：Blue 修 → Red 再攻，按关键词判定 Red 是否放弃，停止原因三选一：`no_more_issue` / `max_rounds` / `red_failed`，每轮都产一条 DPO 对。
>
> 每轮对话天然是一条 DPO/SFT 训练数据。**用不同模型 + 不同 system + 不同温度打破同质化吹捧**；Claude Judge 挂掉自动降级到 reasoner，保证批量产数据时单条失败不拖垮整批。
>
> **定位**：不进主 DAG（对抗不是研究流程的一部分，跑一次 run 就多花一笔对抗 token 不合理）。通过 CLI 子命令 `adversarial-run / adversarial-multi / adversarial-build` 显式触发，产数据落 `data/outputs/dpo_dataset.jsonl`。"

## 11. 🧠 面试必懂的 3 个"认知纠偏"（高频被问懵的点）

### 纠偏 ①：这**不是**论文生成器，是**科研闭环 Agent**

别被 M7 带偏。**论文只是最后的出口形式，不是项目核心价值**。

**类比：一个 AI 博士生做课题**

| 博士生动作 | 对应模块 |
|---|---|
| 和导师澄清题目 | M1 问题精炼 |
| 查 arXiv 看论文 | M2 文献检索 |
| 做笔记整理关系 | M3 知识图谱 |
| 和师兄讨论靠不靠谱 | M4 多 Agent 辩论 |
| 设计实验 | M5 实验方案 |
| 写代码跑实验 | M6 代码沙箱 |
| 写论文 | M7 论文初稿 |
| "换个方向再试试" | M8 分叉回放 |

**一句话定位（升级版，背这个）**：
> "这不是论文生成器，是**模拟科研工作者完整工作流**的 Multi-Agent 系统。核心价值是把'科研'这个模糊过程**结构化 + 可追溯 + 可分叉**，论文只是最后的输出形式。"

---

### 纠偏 ②：M3 之后的流程到底在干嘛（很多人在这里卡住）

**知识图谱不是目的，是中间资产**。流程其实是：

```
M2 拿到 30 篇论文（非结构化）
    ↓
M3 把论文关系结构化成图 +  🔑 发现研究空白
    ↓
M4 基于「用户意图 + 学界现状 + 研究空白」辩论假设是否靠谱
    ↓
M5 假设靠谱 → 设计具体实验
    ↓
M6 跑代码验证
    ↓
M7 写成论文
    ↓
M8 "换个假设" → 从 M4 checkpoint 拉分支重跑
```

**M3 之后每一步都在做"科研决策"**，不是技术堆砌。

---

### 纠偏 ③：研究方向是**谁**决定的？（常被追问）

**错误理解**："M3 发现空白 → 自动决定方向"
**正确理解**：**多 Agent 接力**，分工明确：

| 步骤 | 谁干的 | 怎么干的 |
|---|---|---|
| ① **发现**研究空白 | M3 | 启发式规则：图里"挑战节点"缺对应"方法节点" → research_gaps |
| ② **判断**空白值不值得做 | M4 | 5 Reviewer 辩论 + Meta 终裁，综合用户意图+学界现状+空白 |
| ③ **决定**怎么落地 | M5 | 按 M4 方向设计 datasets/baselines/metrics |

**一句话总结**：
> "**M3 看到哪里是坑，M4 判断坑值不值得跳，M5 决定怎么跳**。空白只是 M4 的输入信号之一，真正的方向决策在 M4。"

**⚠️ 诚实说**：M3 的空白识别是**启发式规则**（ad-hoc 实现），不是高级算法，代码注释自己写了"后续可换成更复杂的子图匹配"。

## 12. 🔥 附录 B 对抗数据工厂——原理深挖（高频追问）

### Q：对抗的是什么？
**A：对抗的是"研究方案文本"，不是 prompt**。Prompt 是静态的"角色人设"（BLUE_SYS、RED_SYS 写死），用来把两个 Agent 塑造成对立立场；真正在博弈的是「方案→攻击→修复」这三段**业务内容**。

### Q：和 GAN 的区别？
**A：借鉴思想但没有反向传播**。模型权重不更新，对抗发生在**纯对话层面**。目的不是训 G 和 D，而是**批量生产 DPO 偏好数据**。

### Q：3 个角色怎么配？

| 角色 | 任务 | 模型 | 温度 | 为什么 |
|---|---|---|---|---|
| Blue | 提方案 & 修漏洞 | deepseek-chat | 0.6 | 创造性 |
| Red | 找 3 个致命漏洞 | deepseek-**reasoner** | **0.8** | 推理强 + 高温加对抗性 |
| Judge | 打分 + 选胜者 | Claude Opus | 0.3 | 关键决策低温稳 |

**关键设计**：Blue/Red **用不同模型 + 不同 system + 不同温度**，打破同质化，避免 LLM 互相吹捧。

### Q：数据流从哪来到哪去？
```
seed_proposals (人工喂 / 理想状态从 M4 通过的方案接)
    ↓
run_round() × N 轮
    ↓
to_dpo_pair() → (prompt, chosen, rejected, score)
    ↓
data/outputs/dpo_dataset.jsonl  ← ⚠️ 到这里断了
    ↓
(未实现) 用 TRL DPOTrainer 微调小模型 → 回灌替代 DeepSeek
```

**诚实说**：**数据生产到 jsonl 就停了，下游微调和回灌闭环没做**。定位是"数据飞轮的上游"，目前是孤岛工具。

### Q：为什么要对抗？不能让 LLM 自己检查吗？
**A：同模型自检有两个硬伤**：
1. **自我吹捧**：倾向给模糊客气的批评，不否定自己
2. **视野盲点**：看不到训练数据外的失败模式

**对抗三层设计强行逼出高质量负面反馈**：
- 换模型（reasoner vs chat）→ 换大脑
- 高温 0.8 → 鼓励发散攻击
- prompt 强制"3 个最致命漏洞" → 不给和稀泥余地
- Judge 用第三方 Claude → 避免阵营偏见

### Q：在整个项目的作用？
**A：分两层说**：
- **代码层面**：graph.py 没 import 它，**目前等于零**，孤岛脚本
- **架构层面**：预期是**数据飞轮的上游**——M4 通过的方案当种子 → 产 DPO 数据 → 微调小模型 → 替代 DeepSeek 降本 / 或回灌 M4 当 Reviewer 提质。**验证了可行性但没跑完整闭环**。

### Q：面试怎么讲（60 秒诚实版）？
> "附录 B 借鉴 GAN 思想做对抗数据生成，但没有反向传播，对抗发生在 prompt 层——Blue 提方案、Red 找漏洞、Blue 修复、Judge 裁决。关键设计是 Blue/Red 用不同模型+不同系统 prompt+不同温度打破同质化。每轮产出一条 DPO 训练对。**目前这是数据飞轮的上游工具，产出到 jsonl 就停了，下游微调闭环没做，属于架构预留扩展点。**"

## 12.5 🔥 M3 升级版：图嵌入 + 链接预测做真正的 Gap Mining

> 被问"如果让你升级 M3 的空白识别"必背。

### 核心思想（一句话）
> **把节点变向量，向量相似度高 + 图里没边 = 研究空白**。
> 类比电商推荐：研究 A 的人通常也研究 B，但你图里 A 和 B 没连 → A→B 就是空白。

### Pipeline 4 步
```
M3 知识图谱 → 图嵌入(node2vec/GCN) → 链接预测 → 扫未连边对 → 输出 gaps
```

### 两种主流技术对比

| 方案 | node2vec | GCN |
|---|---|---|
| 原理 | 图上随机游走当"句子"喂 Word2Vec | 节点聚合邻居信息做加权传播 |
| 用啥信息 | 只用图结构 | 结构 + 节点特征（如论文摘要 BERT） |
| 训练 | 无监督 | 有监督（正样本=真边，负样本=随机不存在的边） |
| 成本 | 5 分钟 CPU | 需要 GPU |
| 适用 | 入门、POC | 生产级 |

### 挖空白的核心算法
```python
for (a, b) in 所有未连边节点对:
    if node_type[a] == 'method' and node_type[b] == 'challenge':
        score = cosine(embed[a], embed[b])
        candidates.append((a, b, score))
candidates.sort(by=score, desc)
return candidates[:top_k]   # 这些就是研究空白
```

输出例子：`('Self-RAG', 'long-doc-QA', 0.91)` → "Self-RAG 在向量空间和长文档 QA 高度相关，但没人做过这个组合"

### 启发式 vs 链接预测（核心差距）

| 维度 | 启发式（项目现状） | 图嵌入 + 链接预测 |
|---|---|---|
| 信号 | 节点入/出度计数 | 图拓扑模式 + 节点语义 |
| 发现范围 | 只能找"被改进很多次的节点" | 任意两节点间潜在缺失关系 |
| 假阳性 | 高（GPT-4 也会被误判） | 低（向量相似度有数学意义） |
| 工程复杂度 | 5 行代码 | 200+ 行 + GPU |

**核心差距**：启发式只回答"哪个节点很热"，链接预测回答"**哪个组合还没人做但理应该做**" —— 后者才是 Gap 的本质。

### 面试话术（高级版）
> "我会换成**图嵌入 + 链接预测**做真正的 Gap Mining：
> 1. 用 node2vec 在 M3 图上做随机游走训嵌入，每个节点 128 维向量；如果有 GPU 升级到 GCN 融入节点的 BERT 特征；
> 2. 训一个 MLP 做链接预测，正样本=真边，负样本=随机采的不存在的边；
> 3. 扫所有未连边的 (method, challenge) 对，按预测概率排序，top-K 就是真正的研究空白。
>
> 本质区别：启发式回答'哪个节点很热'，链接预测回答'哪个组合该有但没有'。后者才是 Gap 的本质。
>
> 我没在项目里做是因为 POC 阶段 ROI 低 —— 下游 M4 辩论会纠偏，且训 GCN 要 GPU。但生产化首选这条路。"

### 一句话记住
> **嵌入把节点变向量，相似度高 + 图里没边 = 研究空白。这才是 Gap Mining 的本质。**

## 12.6 🔥 空白 → Agent：找到研究空白后 M4 具体怎么操作

> 被问"找到空白后 Agent 怎么用"必背。

### 流程链路
```
M3 research_gaps = ["长文档多跳 QA 下 RAG 效果空白", ...]
        ↓
M4 圆桌（空白只是输入之一，不是决策者）
   输入 = M1 PICO + M2 top 论文 + M3 gaps
        ↓
   5 Reviewer 并行评审 → Meta-Reviewer 终裁
        ↓
   meta_decision = {
     decision: "accept" | "revise" | "reject",
     final_rating: 7.5,
     key_directions: "聚焦多跳 QA"
   }
```

**关键澄清**：Agent **不是只看空白做决定**，而是综合三份输入辩论。空白只是"候选课题"，靠不靠谱要 5 个 Reviewer 投票。

### 4 步操作

**Step 1 · 准备上下文**（critique_node 函数）
```python
refined_q       = M1 的精炼问题
method_summary  = M1 的 PICO + M3 的 gaps
top_papers      = M2 的 top-5 论文
```

**Step 2 · 5 个 Reviewer 并行评审**（asyncio.gather）

| Reviewer | 看 gaps 的视角 |
|---|---|
| novelty | "这 gap 真没人做过吗？查图" |
| methodology | "针对 gap，PICO 设计合理吗" |
| statistics | "样本量、检验方法靠谱吗" |
| reproducibility | "数据集和代码可获得吗" |
| devil | "这 gap 其实是死胡同，理由 X" |

每人输出一张 CritiqueCard：`{reviewer, rating 1-10, strengths, weaknesses, rationale}`

**Step 3 · 方差检测 → 二次辩论**
```python
ratings = [7, 8, 3, 7, 8]
variance = pvariance(ratings)  # 4.16 > 2
→ 触发 Devil's Advocate Round 2:
  把 Round 1 摘要喂给 devil，让它看分歧后再评一次
```

**Step 4 · Meta-Reviewer 终裁**（Claude Opus）
```json
{
  "decision": "accept_with_revision",
  "final_rating": 7.2,
  "key_weakness": "缺多跳 QA 评测",
  "next_action": "在 M5 增加 HotpotQA"
}
```
- `accept` → 进 M5 设计实验
- `reject` → 回 M1 重新精炼问题

### 面试话术（完整版）
> "M3 找到的空白作为输入之一进 M4。M4 是 5 个人设 Reviewer 并行评审 —— novelty 查这 gap 真没人做过吗，methodology 查方法设计对不对，devil 专门唱反调。每个 Agent 输出一张 CritiqueCard 含 rating 和 weaknesses。如果 5 个 rating 方差 >2 说明分歧大，触发 Devil's Advocate 第二轮看着 Round 1 摘要再评一次。最后 Claude Opus 当 Meta-Reviewer 综合所有卡片出 decision —— accept 进 M5 设计实验，reject 回 M1 重新精炼。**真正的方向决策不在 M3 的启发式空白识别，而在 M4 的多 Agent 辩论**。"

### 一句话记住
> **5 个角色 LLM 并行评审，方差大触发二次辩论，Claude 终裁出方向。空白是输入信号，Agent 是决策器。**

## 12.7 🔥 对抗数据的"产地、长相、去向"（附录 B 终极版）

> 被问"对抗数据哪来的、长啥样、训什么模型"必背。

### 1️⃣ 产地：`appendix/adversarial/red_blue.py`

**通过 CLI 显式触发**（不在主 DAG，避免每次 run 都产生对抗 token 成本）：
```bash

```

**一轮流程**（1 个种子 = 3 次 LLM 调用 = 1 条数据）：
```

  ↓
Red (deepseek-reasoner, 温度 0.8) 找 3 个致命漏洞
  ↓
Blue (deepseek-chat, 温度 0.6) 看到攻击后修复
  ↓
```

```python
trainer = DPOTrainer(
    model=AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-7B-Instruct"),
```

**DPO vs SFT**：SFT 只学 chosen 不学 rejected；DPO 同时用两者学**偏好边界**，对齐效果更好且比 RLHF 稳定。

## 12.8 🔥 RAG vs GraphRAG 区别（高频必问）

### 一句话对比
> **RAG 检索文本片段，GraphRAG 检索子图 + 社区摘要**。
> RAG 给模型一堆文档碎片，GraphRAG 给模型结构化的实体关系网。

### 核心对比表

| 维度 | RAG | GraphRAG |
|---|---|---|
| 索引对象 | 文本 chunks | 实体 + 三元组 + 文本 |
| 检索单位 | Top-K 文档片段 | Top-K 子图（实体+邻居+关系） |
| 召回方式 | 向量相似度 | 图遍历 + 向量 + 社区摘要 |
| 存储 | 向量库（Qdrant） | 图库（Neo4j）+ 向量库 |
| 适合场景 | 单点问答、事实查询 | 多跳推理、全局归纳 |
| 构建成本 | 低（切块 + embedding） | 高（LLM 抽三元组） |
| 可解释性 | 弱 | 强（能追溯图结构） |

### GraphRAG 流程（微软 2024）
```
离线构建:
  文档 → LLM 抽三元组 → 建图 → Leiden 聚类 → LLM 生社区摘要 → 存 Neo4j
查询:
  Local Search:  问具体实体 → 拿实体+邻居+关联 chunks
  Global Search: 问领域总览 → 社区摘要 map-reduce
```

### 4 个核心差异
1. **检索粒度**：RAG 平铺 chunks；GraphRAG 结构化子图
2. **多跳推理**：RAG 靠 LLM 自己拼（易错）；GraphRAG 图遍历天然支持
3. **全局问题**："这领域有哪些流派" RAG 答不了，GraphRAG 社区摘要专治
4. **构建成本**：RAG 1 小时，GraphRAG 要数小时到数天

### 在本项目的位置
| 模块 | 技术 |
|---|---|
| M2 文献检索 | 纯 RAG：Query Rewriting + RRF + 多源向量 |
| M3 知识图谱 | GraphRAG 的**构建**部分：LLM 抽三元组建图 |
| M4 批判辩论 | 吃 M2 + M3 组合结果 |

**⚠️ 诚实说**：只做了 GraphRAG 的**构建和简单查询**，**没做 Leiden 社区聚类、没做社区摘要**。严格说是"KG-enhanced RAG"不是完整 GraphRAG。

### 面试话术
> "核心区别是检索对象不同：RAG 检索文本片段拼到 prompt 喂 LLM，适合单点事实；GraphRAG 检索知识图谱子图 —— 先 LLM 抽三元组建图，再 Leiden 聚类生社区摘要，查询时图遍历拿子图+摘要一起喂 LLM。优势在**多跳推理**（图遍历天然支持）和**全局归纳**（社区摘要能答'这领域有哪些流派'）。代价是构建贵一个数量级。我项目 M2 是纯 RAG，M3 做了 GraphRAG 的构建部分，但**社区聚类和摘要没做**，严格说是 KG-enhanced RAG。"

### 追问预案
| Q | A |
|---|---|
| 微软 GraphRAG 社区摘要怎么做？ | Leiden 分层聚类 → 每社区 LLM 生摘要 → map-reduce 汇总 |
| 什么时候该用 GraphRAG？ | 实体关系密集 + 多跳刚需 + 全局归纳需求，三者之一 |
| GraphRAG 缺点？ | 构建贵、抽取质量靠 LLM、更新成本高 |
| 混合检索怎么做？ | 图召回 + 向量 + BM25，RRF 融合 |

### 一句话记住
> **RAG 检索片段，GraphRAG 检索子图+社区摘要。前者单点事实，后者多跳+全局。代价是构建贵 10 倍。**
